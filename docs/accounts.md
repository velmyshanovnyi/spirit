# Акаунти: створення, backup, мультипристрій

## Типи акаунтів

- **Ефемерний "світ"** — identity-keypair генерується як `extractable: false`, живе лише в пам'яті вкладки. Backup технічно неможливий і не пропонується — це свідома властивість режиму, а не обмеження. Одностроєвий за визначенням.
- **Перманентний профіль** — identity-keypair генерується як `extractable: true`, зберігається в IndexedDB, зашифрований ключем, похідним від passphrase користувача (PBKDF2/Argon2 → AES-GCM). Підтримує backup і мультипристрій.

## Створення акаунта (UX-флоу)

1. Стартовий вибір: **"Швидкий чат"** (ефемерний, нуль кроків налаштування) або **"Створити профіль"** (перманентний).
2. Для перманентного профілю: генерація identity-keypair + пропозиція локального passphrase (шифрує IndexedDB, не є backup-секретом сам по собі).
3. Крок **backup** — не блокує використання застосунку, але настійно рекомендований:
   - Вибір формату: **мнемонічна фраза**, **зашифрований keyfile**, або обидва одразу.
   - Опція **"Пропустити"** — дозволена, з явним попередженням: "Без резервної копії втрата пристрою/даних браузера = незворотна втрата акаунта". У такому разі в UI лишається постійний, ненав'язливий банер-нагадування, який зникає тільки після виконання backup (можна зробити пізніше з налаштувань профілю).

## Backup: один секрет, два кодування

Мнемоніка і keyfile — це **не два різні механізми відновлення**, а два способи записати той самий секрет, щоб не подвоювати криптографічну поверхню й не потребувати детермінованої деривації EC-ключів (яка вимагала б ручної роботи з еліптичною кривою поза `SubtleCrypto`):

1. Identity-keypair генерується звичайним `crypto.subtle.generateKey` (як завжди, випадково).
2. Приватний ключ експортується в сирі байти: `crypto.subtle.exportKey("pkcs8", privateKey)`.
3. **Мнемонічна фраза**: ці байти (32 байти ентропії приватного скаляра P-256) кодуються як BIP39-подібний word list — 24 слова, з контрольною сумою (SHA-256 перших біт). Це чисте кодування ентропії в слова, без прив'язки до якоїсь конкретної крипто-мережі (BIP39-механізм переюзаний як zero-dependency encoding, а не для деривації HD-гаманця).
4. **Keyfile**: ті самі байти, зашифровані passphrase користувача (PBKDF2 → AES-GCM), збережені як завантажуваний JSON-файл.
5. Відновлення: користувач вводить мнемоніку АБО завантажує keyfile + passphrase → отримуємо ті самі сирі pkcs8-байти → `crypto.subtle.importKey` → identity-keypair відновлено повністю (та сама публічна ідентичність, той самий пул контактів, що впізнають цей ключ).

Наслідок: не потрібно імплементувати SLIP-10/детерміновану EC-деривацію — обидва формати backup є прямим кодуванням реально згенерованого ключа.

## Мультипристрій (device linking)

Централізованого серверу для трекінгу пристроїв немає, тож мультипристрій будується на **делегованих сертифікатах пристроїв**, а не на серверній таблиці "user → devices":

1. Кожен пристрій, приєднаний до профілю, генерує власний **device keypair** (ECDSA P-256), окремий від identity-keypair.
2. Приєднання нового пристрою (device linking):
   - На новому пристрої користувач обирає "Приєднати цей пристрій до існуючого акаунта".
   - Новий пристрій генерує device keypair і публікує запрошення на приєднання через ту саму інфраструктуру invite-токенів, що й звичайний чат (див. [signaling-protocol.md](signaling-protocol.md)) — тобто мультипристрій використовує вже наявний P2P + E2EE стек, не новий канал.
   - Існуючий (уже авторизований) пристрій встановлює P2P E2EE з'єднання з новим, підписує публічний ключ нового пристрою своїм **identity private key** → отримуємо **device certificate**: `sign(identity_priv, device_pubkey + timestamp)`.
   - Existing-пристрій передає новому: device certificate, копію identity private key (спрощена MVP-модель — див. нижче), поточний список контактів і (опційно) знімок історії — усе через щойно встановлений E2EE-канал, ніколи через сигнальний вузол.
3. Контакти приймають повідомлення від будь-якого пристрою, що показує валідний device certificate під відомим identity public key.

### Компроміс MVP: спільний identity private key між пристроями

Для простоти MVP кожен зв'язаний пристрій отримує повну копію identity private key (а не тільки делегований device key з обмеженими правами). Це означає:
- Компрометація будь-якого одного пристрою розкриває identity private key повністю.
- Відкликання пристрою (видалення зі списку зв'язаних) не "скасовує" те, що цей пристрій уже знав ключ — воно лише сигналізує контактам більше не довіряти сертифікату цього пристрою надалі.

Це задокументований trade-off (див. [decisions.md](decisions.md), D9). Строгіша схема з делегованими правами без розкриття identity-ключа іншим пристроям — Фаза 4.

## Синхронізація між пристроями (eventually-consistent, без сервера)

Без централізованого relay неможливо гарантувати миттєву доставку на всі активні пристрої одночасно кожного разу. Модель для MVP:

- Вхідне повідомлення доставляється тому пристрою, який у цей момент відповідає на P2P-сесію (тобто "активний" пристрій для даного контакту в даний момент).
- Пристрої користувача періодично (при одночасному онлайні) встановлюють прямий P2P-канал **між собою** (той самий сигнальний+E2EE стек, автентифікація через device certificates) і звіряють/синхронізують історію та список контактів.
- Це навмисно eventually-consistent, а не real-time fan-out на всі пристрої — компроміс, прийнятний для serverless-архітектури (див. D9).

## Відкликання пристрою

Профіль підтримує список активних device certificates. Відкликання = профіль підписує оновлений список (виключаючи відкликаний пристрій) новим identity-підписом; при наступному P2P-хендшейку контакти оновлюють свій кеш дозволених device certificates для цієї identity і перестають приймати повідомлення від відкликаного сертифіката.
