# OAuth-верифікація (Google, Yandex, Telegram, Facebook)

## Місце в архітектурі

Це **не** заміна анонімного identity-ключа (D3 з [decisions.md](decisions.md) лишається чинним) і не окремий механізм авторизації в Spirit — це автоматизована, криптографічно сильніша версія вже спроєктованого [identity-verification.md](identity-verification.md). Замість того, щоб користувач вручну публікував підписаний proof-текст на зовнішній платформі, OAuth-провайдер видає **вже підписаний ним самим** токен (JWT), який Spirit прив'язує до свого identity-ключа й перевіряє повністю на клієнті — без бекенду Spirit, для провайдерів зі стандартним OIDC.

Ключова відмінність від URL-доказів: OAuth-атестація — це **разова, point-in-time подія** ("підтверджено через Google email@gmail.com станом на дату X"), а не URL, який можна перефетчити пізніше для переперевірки (сам ID-токен короткоживучий і одноразовий). Бейдж показує дату останньої атестації; освіжити його — повторний OAuth-логін, не автоматичний recheck.

## Модель: nonce-прив'язка (без бекенду Spirit)

Стандартний OIDC-флоу вже має механізм саме для цього: параметр `nonce` у запиті авторизації повертається незмінним усередині підписаного ID-токена. Spirit використовує це так:

1. Профіль → "Підтвердити через Google" → клієнт обчислює `nonce = fingerprint(identityPublicKey)` (той самий fingerprint, що й у [identity.js](../client/js/identity.js)).
2. Клієнт ініціює OIDC-запит до Google з цим `nonce` (через Google Identity Services JS SDK, popup/redirect — без client secret, це публічний SPA-флоу).
3. Google повертає підписаний ID-токен (JWT), що містить `sub` (Google user ID), `email`, `nonce` (наш fingerprint), `aud` (client_id), `iss`, `exp`.
4. Клієнт перевіряє підпис JWT **локально**, використовуючи публічний JWKS Google (`https://www.googleapis.com/oauth2/v3/certs`, доступний з CORS `*`, знайдений через OIDC discovery `https://accounts.google.com/.well-known/openid-configuration`) — жодного проксі чи `fetch_proof` не потрібно, на відміну від URL-доказів.
5. Клієнт перевіряє: `nonce` збігається з fingerprint власного identity-ключа, `aud` збігається з налаштованим client_id, `iss === "https://accounts.google.com"`, токен не протермінований.
6. Успіх → створюється підписаний identity-ключем attestation-запис, що додається до того самого proof-set з [identity-verification.md](identity-verification.md) (розширення структури, не новий механізм):
   ```json
   { "type": "oauth", "provider": "google", "email": "user@gmail.com", "verified_at": 1770000000 }
   ```
7. Той самий proof-set (з URL-доказами і OAuth-атестаціями впереміш) доставляється контактам тим самим шляхом — підписаний identity-ключем блок під час P2P-хендшейку (D13).

## Google — готово до реалізації зараз

- SDK: Google Identity Services (`accounts.google.com/gsi/client`), публічний, не вимагає server-side client secret для ID-токен флоу.
- Верифікація JWT: стандартна (RS256), публічний ключ через JWKS — все клієнтське, немає потреби навіть у `fetch_proof`, бо Google's JWKS endpoint сам віддає CORS `*`.
- Потрібен лише публічний Google OAuth Client ID (налаштовується в Google Cloud Console), не секрет — можна закомітити в конфіг клієнта, це публічна інформація за дизайном OIDC public-client флоу.

## Yandex — той самий патерн, з одним нез'ясованим нюансом

- Yandex OAuth історично базується на implicit-грант (`access_token` у фрагменті URL), не завжди строго OIDC-сумісний ID-токен з `nonce`.
- Потрібно перевірити перед імплементацією: чи Yandex ID (сучасніша версія) підтримує повноцінний OIDC ID-токен з `nonce`, чи доведеться робити додатковий виклик `https://login.yandex.ru/info` з `access_token` для отримання даних користувача — і чи цей endpoint віддає CORS-заголовки для прямого клієнтського виклику, чи знадобиться `fetch_proof`-подібний проксі. **Позначено як відкрите технічне питання для спек-фази Yandex-інтеграції.**

## Telegram, Facebook — заплановано, вимагають серверного компонента

Обидва не мають чистого клієнтського флоу без секрету:

- **Telegram Login Widget** повертає дані, підписані HMAC-SHA256 з **bot-токеном**, який має лишатись на сервері (ніколи в браузері) — інакше будь-хто зможе підробити дані входу.
- **Facebook Login** технічно має клієнтський JS SDK, що видає `access_token`, але повноцінна верифікація його автентичності (виклик `debug_token`) вимагає app secret на сервері.

Для обох — план: опційний endpoint на сигнальному вузлі (аналогічно `fetch_proof`, D12) — оператор явно вмикає, тримає bot-токен/app-secret у своєму конфізі, ніколи не передає клієнту. Деталі — окрема специфікація, коли дійде черга (Фаза 2d, після Google/Yandex).

## Безпекові інваріанти

- OAuth-провайдер **ніколи** не замінює identity-ключ — лише додає опційний, відкличний (D13) атестаційний запис поверх нього.
- `nonce`-прив'язка гарантує: атестація дійсна лише для того identity-ключа, для якого її запитували (неможливо взяти чужий Google-логін і приліпити до свого ключа).
- Компрометація OAuth-акаунта користувача (наприклад, хтось інший увійшов у чужий Google-акаунт) дозволяє створити фальшиву атестацію "верифіковано як email X" — це той самий рівень довіри, що й будь-який OAuth "Sign in with X" у будь-якому застосунку; Spirit не підвищує і не знижує цю довіру, лише передає її через identity-verification.md механізм.
