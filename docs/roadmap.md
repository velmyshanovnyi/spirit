# Roadmap

## Фаза 0 — Архітектура (поточна)

Документація в `docs/`: компоненти, протокол сигналінгу, дизайн E2EE, журнал рішень. Коду немає.

## Фаза 1 — MVP ✅ ЗАВЕРШЕНО

Повна специфікація та історія review — [specs/phase1/mvp.md](../specs/phase1/mvp.md).

- Клієнт (Секції 1-5, test-first, Vitest): identity/ECDH keygen, E2EE (ECDH → HKDF → AES-256-GCM), signaling client, WebRTC-оркестрація, мінімальний UI. `client/`.
- Сигнальний вузол (Секції 6-10, PHP, верифіковано живим деплоєм на `spirit.kibr.com.ua` — PHP/Composer недоступні локально, explicit test-first waiver): файлове сховище з атомарним записом і GC, invite-токени, CORS, rate-limiting, повний контролер дій + опційний `fetch_proof` (SSRF-захищений). `server/`.
- Живо верифіковано: повний P2P-хендшейк, одноразовість invite-токенів під конкурентним навантаженням (3/3 рази), коди помилок, CORS preflight, rate limiting, SSRF-блокування (decimal/hex/octal/IPv6-кодування).
- Персистентність: відсутня — суто ефемерна сесія в пам'яті вкладки (за задумом MVP).

## Фаза 2 — Профілі, backup та мультипристрій ✅ ЗАВЕРШЕНО

Специфікації та історія review — [specs/phase2/profiles.md](../specs/phase2/profiles.md) (Секції 1-10) і [specs/phase2/history-sync-accounts.md](../specs/phase2/history-sync-accounts.md) (Секції 11-15). Test-first, 221 тест у сумі по проєкту.

- IndexedDB-обгортка (`db.js`) + passphrase-шифрування сховища (PBKDF2-HMAC-SHA256 600k → AES-256-GCM, `vault.js`).
- Перманентні профілі (`profile.js`): створення/розблокування, non-extractable ключ після завантаження; **мультиакаунти** (записи за fingerprint-ключами, селектор у UI, лінива міграція legacy-запису).
- Backup identity-ключа: BIP39-подібна мнемоніка (24 слова) та зашифрований keyfile — обидва з відновленням; UI-флоу з backup-кроком.
- Мультипристрій (`deviceLinking.js`): device-сертифікати, приєднання нового пристрою через P2P/E2EE-канал, відкликання через версійований підписаний список (анти-replay монотонність), транспорт списку контактам у чат-хендшейку.
- **Автентифікований identity-announce** (`identityAnnounce.js`): TOFU-контакти (`contacts.js`), прив'язка до session-ECDH (анти-replay/reflection/MITM-transplant), гейтування вхідного тексту на верифіковану identity.
- **Зашифрована історія повідомлень** (`historyStore.js`): AES-GCM під vault-ключем, per-profile/per-contact неймспейс, рендер історії при верифікації відомого контакта.
- Онбординг-UX ([migration.md](migration.md)) — перенесено в наступні ітерації; латентне питання: per-profile scoping контактів (див. review Секції 15).

### Фаза 2b — Опційний імпорт (не блокує Фазу 2) ✅ ЗАВЕРШЕНО

Специфікація та історія review — [specs/phase2b/import.md](../specs/phase2b/import.md) (Секції I1-I3).

- Чисті парсери (`importParsers.js`): `parseContactList`/`parseChatExport` для Telegram JSON, vCard, WhatsApp (контакти й `.txt`-історія).
- Importer-модуль для контактів: pending-стор `importedContacts.js` (окремий від `contacts.js`, без fingerprint до матчингу), UI на екрані «Контакти», ВИКЛЮЧНО ручна верифікація збігів (ніякого авто-матчингу за іменем/номером).
- Importer-модуль для історії переписки: розпарсені повідомлення тримаються як `pendingMessages` того ж pending-запису до матчингу; після ручного підтвердження — запис у `historyStore.js` з `imported: true` (без зміни схеми стору), візуальна мітка "історичне (імпортоване)" в чат-лозі.
- Деталі — [migration.md](migration.md).

### Фаза 2c — Опційна верифікація зовнішньої ідентичності ✅ ЗАВЕРШЕНО

Специфікація та історія review — [specs/phase2c/identity-verification.md](../specs/phase2c/identity-verification.md) (Секції A-E, plus живий тест fetch_proof на обох вузлах).

- Генерація/публікація/перевірка URL-доказів (Telegram, вебсайт, GitHub gist тощо), бейджі підтвердження в UI контакту.
- Proof-set: доставка через P2P-хендшейк, періодична переперевірка (24-год таймер + ручна кнопка), явне відкликання власником.
- `fetch_proof` proxy-дія на сигнальному вузлі — SSRF-захищена, реалізована й **увімкнена** на kolomedi/kibr (`server/config.secrets.php`, не в git).
- Email/DKIM-доказ — не реалізовано, можливе майбутнє розширення.
- Деталі — [identity-verification.md](identity-verification.md).

### Фаза 2d — OAuth-верифікація (Google/Yandex) ✅ ЗАВЕРШЕНО (Telegram/FB — заплановано)

Специфікація — [specs/phase2d/google-oauth.md](../specs/phase2d/google-oauth.md).

- Автоматизована, криптографічно сильніша версія [identity-verification.md](identity-verification.md): замість ручної публікації proof-тексту, підписаний провайдером ID-токен (JWT) прив'язується до Spirit-ключа через `nonce`.
- Google: чистий клієнтський OIDC-флоу (без бекенду Spirit) реалізовано — деталі в [oauth-verification.md](oauth-verification.md).
- Yandex, Telegram, Facebook: не реалізовано; Telegram/Facebook вимагають server-side перевірки (bot-токен/app-secret не можна тримати в браузері) — заплановано як опційний endpoint на сигнальному вузлі, за тією ж моделлю підвищеного ризику, що й `fetch_proof`.
- Не замінює анонімний identity-ключ (D3 лишається чинним) — виключно опційний шар "визнаності" поверх нього.

## Фаза 3 — Портативні акаунти, ефемерний режим, редизайн UI ✅ ЗАВЕРШЕНО

Специфікації та історія review — [specs/phase3/deterministic-accounts.md](../specs/phase3/deterministic-accounts.md) (Секції H1-H4) і [specs/ui/ephemeral-spirit-mode.md](../specs/ui/ephemeral-spirit-mode.md) (Секції F1-F5, G1-G2).

- **Крос-серверні (портативні) акаунти**: `identity = Argon2id(password, salt=name)` через vendored `hash-wasm` (WASM), логін `spirit<name10><tail16>` публічний, будь-який незалежний вузол відтворює той самий keypair без локального сховища. Живо перевірено крос-хостово (kolomedi → kibr, ідентичний Spirit ID).
- **Ефемерний "режим духів"**: один клік — авто-ідентичність + анонімний нік ("Тихий Привид") + авто-створення кімнати + авто-перехід у чат; приєднання за invite-посиланням — нуль кліків.
- Екран «Акаунт»: чіткий тумблер create/login (не два блоки одночасно), MRU-список останніх 10 акаунтів (браузер-рівень).
- Центрування одно-карткових екранів на десктопі.

## Фаза 4 — Розширення ✅ ЗАВЕРШЕНО

- ~~Автентифікований обмін ключами (підпис ECDH-ключа identity-ключем) замість ручного safety number~~ — **вже реалізовано** (Фаза 2, Секція 12, `identityAnnounce.js`; `docs/e2ee.md` та цей запис виправлені 2026-07-18 — раніше помилково вважалось невиконаним, документація була застарілою, коду це не стосувалось).
- ~~Групові чати~~ — **✅ ЗАВЕРШЕНО** (`specs/phase4/group-chats.md`, Секції GC0-GC3). Узгоджено з користувачем (AskUserQuestion, 2026-07-18): mesh-архітектура на базі N попарних P2P-з'єднань (без нового спільного групового ключа), лише текстовий чат у першому циклі (без групових дзвінків). GC0: `state.peers` — повний multi-connection рефакторинг `state` (одиночні поля `pc`/`channel`/`sessionKey`/... стали проксі-парою на активний запис `Map`), найризикованіша секція, 2 ітерації review (1 LOW async-race знахідка виправлена). GC1: `client/js/groups.js` — сховище груп (`createGroup`/`getGroup`/`listGroups`/`updateGroupMembers`/`deleteGroup`), `historyStore.js` без змін (`groupId` як звичайний `contactId`-рядок). GC2: оркестрація приєднання — **свідома звужена реалізація, не повний mesh** (продуктове рішення прийняте автономно 2026-07-18 через відсутність presence-детекції, zero-database): кожен учасник підключений щонайменше до того, хто його запросив (топологія зірка/дерево запрошень), не гарантовано "кожен з кожним"; `group-member-joined` control-повідомлення — best-effort сповіщення інших підключених членів. GC3: `group`-параметр в invite-URL (замикає прогалину GC2, де тегувала лише сторона запрошувача — тепер обидві сторони); `sendGroupMessage` — фанаут-надсилання (шифрування per-канал, статичний session key, БЕЗ ratchet, той самий прецедент що й передача файлів); новий control-тип `group-message` з anti-spoofing-гейтом (заявлений `groupId` звіряється з тим, чим ФАКТИЧНО тегове з'єднання); UI "Мої групи" переюзує наявний `#chat-log`-подібний конверсаційний екран через `state.activeGroupId`-routing замість окремого екрана; попутний фікс (exec review iter1) — серіалізація обробки вхідних повідомлень через `state.messageDispatchLock`, необхідна щойно з'явились справжні одночасні з'єднання. 670/670 тестів зелені (з 640 на старті GC0). **Обсяг, що НЕ реалізовано в цьому MVP**: повна mesh-топологія "кожен з кожним" (можливе майбутнє розширення GC2), групові дзвінки/відео.
- ~~Передача файлів через DataChannel~~ — **✅ ЗАВЕРШЕНО** (`specs/phase4/file-transfer.md`, Секції FT1+FT2). FT1: чисте ядро чанкування/base64/SHA-256/збирання (`client/js/fileTransfer.js`), окремий exec review. FT2: інтеграція в чат-потік — control-повідомлення `file-offer`/`file-accept`/`file-reject`/`file-chunk` на статичному `sessionKey` (той самий шлях, що й інші control-повідомлення, не ratchet-ланцюжок), явне прийняття/відхилення перед передачею жодного чанка, backpressure через `bufferedAmount`/`bufferedamountlow`, SHA-256-перевірка цілісності після збирання (розбіжність = явна помилка, файл не пропонується як завершений), UI-попередження при >100MB без жорсткого ліміту. Той самий гейт верифікації peer, що й для звичайного тексту чату. 640/640 тестів зелені, exec review зійшовся на ітерації 1 з двома задокументованими (не виправленими) вузькими обмеженнями першої ітерації: банер підтримує лише одну вхідну пропозицію одночасно, і backpressure-очікування на канал спільне для одночасних вихідних передач.
- ~~UI для одночасного використання кількох сигнальних/TURN вузлів~~ — **✅ ЗАВЕРШЕНО** (`specs/phase4/multi-node-ui.md`). Найменший обсяг Фази 4 — чисто UI/`localStorage`-фіча над наявними полями `server-url`/`stun-url`/`force-turn-relay`, без змін до протоколу чи `signalingClient.js`/`server/`. Ключ `spirit.signalingNodes` (той самий рівень сховища, що й `spirit.theme`/`spirit.locale` — налаштування пристрою, не акаунта): список збережених вузлів на екрані «Сервер», кнопка "Зберегти поточні налаштування як...", клік на запис заповнює три поля (без авто-перепідключення — узгоджено з наявною ручною філософією цих полів), видалення запису. Порожній список (типовий стан для всіх наявних користувачів) не змінює наявну поведінку — чиста адитивна фіча. Exec review (1 ітерація, зійшлося): одна знахідка (відсутній shape-guard для елемента списку з неповною формою) виправлена інлайн. 676/676 тестів зелені (з 670 на старті). Цим пунктом закривається весь бектрог Фази 4 (групові чати, передача файлів, тепер і мульти-вузлова UI).

**TOFU при першій зустрічі з контактом — ✅ ЗАВЕРШЕНО (мінімальний scope) 2026-07-18.** Автентифікований обмін ключами (вище) захищає ECDH-сесію ПІСЛЯ того, як identity-ключ контакту вже прийнятий, але сам момент першого прийняття identity-ключа не мав позаканального доказу за замовчуванням — лише TOFU, і наявні proof-механізми (2c/2d) лишались опційними й непомітними. Узгоджено з користувачем (AskUserQuestion): посилити UI-помітність наявних механізмів, без нової криптографії. Реалізовано: контакт без жодного proof тепер явно позначається бейджем "не підтверджено" в списку контактів (`client/js/app.js`'s `renderContactsScreen`, `contacts.unverified` i18n-ключ у всіх 11 локалях) — статус видимий постійно в списку, не лише в момент першої зустрічі (де вже є safety-number-хінт з П4). Test-first, 2 нових тести. Подальше UI/UX-розширення (наприклад, клікабельний CTA прямо з бейджа до екрана додавання proof) лишається опційним майбутнім поліпшенням, не блокує.

## Фаза 5 — Безпека та розподіленість ✅ ЗАВЕРШЕНО

Детальний бектрек, рішення й повна історія review — [specs/phase5/security-hardening.md](../specs/phase5/security-hardening.md), з окремими розгорнутими специфікаціями для найбільших секцій: [push-notifications.md](../specs/phase5/push-notifications.md), [social-recovery.md](../specs/phase5/social-recovery.md), [sybil-resistance.md](../specs/phase5/sybil-resistance.md), [metadata-resistant-transport.md](metadata-resistant-transport.md).

- **P1 — Метадані-стійкий транспорт**: (а) UI-перемикач "форсувати TURN-релей" (`iceTransportPolicy:'relay'`), реалізовано кодом; (б) власний STUN/TURN на kolomedi/kibr — технічно неможливий на поточному shared-хостингу (немає SSH/довготривалих процесів), закрито дослідженням, рекомендовано сторонній TURN через те саме поле; (в) Tor-транспорт для сигналінгу — `.onion`-URL вже підтримується без змін коду (звичайний `fetch()` без protocol-фільтра).
- **P2 — Ratchet поверх ECDH-хендшейку**: мінімальний симетричний KDF-chain ratchet (forward secrecy в межах сесії, без повного Double Ratchet) для user-typed чат-повідомлень; control-повідомлення свідомо лишились на статичному session key (узгоджено з користувачем).
- **P3 — Encrypted push notifications**: RFC 8291/8292 Web Push з нуля (без vendored-бібліотек), VAPID, Service Worker, P2P-обмін підпискою між верифікованими контактами (сервер підписку не бачить), надсилання при спробі писати офлайн-контакту — "дзвінок у двері", не доставка пропущених повідомлень (zero-database, D1, непорушний). Живою перевіркою виявлено й виправлено production-баг: Service Worker реєструвався без `{type:"module"}`, ламаючи весь функціонал сповіщень.
- **P4 — Соціальне відновлення (Shamir secret sharing)**: розщеплення 32-байтного identity-скаляра між довіреними контактами (P2P + QR/текстовий канал), поріг k-з-n, окреме сховище `trustedShares` на боці довіреної особи, повний флоу відновлення на екрані «Акаунт». Також у цій фазі: **safety-number UI-підказка** (раніше прихована, тепер персистентна при першій зустрічі з контактом) і **Sybil-стійкість** через proof-of-work на `create_invite` (hashcash-подібний, прив'язаний до часового вікна+identity, anti-replay). Живою перевіркою виявлено й виправлено два production-баги: `solvePow` зависав на 30+с через послідовний `await` замість батчингу, і рекомендована складність 20 біт була нереалістично повільною навіть після фіксу — перекалібровано на 16 біт за реальними вимірюваннями.
- Соціальне відновлення й Sybil-стійкість підтверджують той самий урок, що вже виявила Секція P3: unit-тести (Node/vitest) і exec-review не бачать реальної продуктивності/поведінки Web Crypto й Service Worker у справжньому браузері — обов'язкова жива перевірка реального UI-флоу після кожного деплою, не лише тестів, лишається чинним правилом проєкту.

## Бектрог UI (записано за прямим запитом користувача, 2026-07-22, не почато)

- **Режим редагування дизайну**: кожен інтерфейсний блок (сайдбар, тулбар, плаваюче відео, чат) користувач може рухати/переміщувати/налаштовувати на свій смак. Активований режим → можна редагувати розташування; вимкнений → відображається з збереженими користувацькими налаштуваннями (localStorage, той самий принцип, що й spirit.floatingVideoRect/spirit.folders). Кнопка "Скинути до дефолтних значень" повертає початковий layout.
