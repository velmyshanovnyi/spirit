# Архітектура

## Компоненти системи

### 1. Клієнт (браузер)
Автономна веб-сторінка (HTML5 + Vanilla JS, ES Modules). Не має серверного бекенду для себе — може лежати на будь-якому статичному хостингу або відкриватись локально.

- **Identity-модуль** — генерація та зберігання криптографічних ключів через Web Crypto API.
- **Signaling-клієнт** — HTTP-запити (long-polling) до обраного користувачем PHP-вузла.
- **WebRTC-модуль** — `RTCPeerConnection` + `RTCDataChannel`, встановлення прямого з'єднання пір-до-піра.
- **E2EE-модуль** — узгодження сесійного ключа та шифрування/дешифрування кожного повідомлення (див. [e2ee.md](e2ee.md)).
- **Сховище (Фаза 2)** — IndexedDB для зашифрованої історії та мультиакаунтних профілів.

### 2. Сигнальний вузол (PHP)
Ультралегкий скрипт без зовнішніх залежностей, який лише допомагає двом клієнтам обмінятись SDP/ICE даними для встановлення WebRTC-з'єднання. Не бачить і не зберігає жодного вмісту переписки.

- Файлове сховище (`database.json`) замість СУБД, з `LOCK_EX` для уникнення race conditions.
- Самоочищення застарілих сесій та інвайтів за TTL (garbage collection при кожному запиті).
- Керування доступом через одноразові invite-токени (див. [signaling-protocol.md](signaling-protocol.md)), опційно — статичний whitelist для приватних вузлів.
- Rate-limiting/anti-flood на рівні файлового лічильника per-IP.

Вузол — замінний і не є єдиною точкою довіри: користувач може підняти власний вузол або обрати чужий публічний. Компрометація вузла розкриває метадані з'єднання (хто з ким намагався зв'язатись, коли), але не вміст переписки завдяки E2EE.

### 3. P2P-транспорт (WebRTC)
Пряме з'єднання `RTCDataChannel` між клієнтами після успішного SDP/ICE handshake. DTLS шифрує канал від пасивного мережевого спостерігача, але не захищає від сигнального вузла, який бачить SDP/ICE під час встановлення з'єднання — тому потрібен окремий E2EE-шар на прикладному рівні.

## Потік встановлення з'єднання

```
Ініціатор (A)                     Сигнальний вузол (PHP)                Учасник (B)
     |                                     |                                  |
     |-- generate identity + ECDH keys     |                                  |
     |                                     |                                  |
     |-- create_invite ------------------->|                                  |
     |<-- invite_token, room_id -----------|                                  |
     |                                     |                                  |
     |   (передає invite_token поза каналом: лінк / QR-код)  -------------->  |
     |                                     |                                  |
     |-- create_offer(room_id, sdp,        |                                  |
     |    ecdh_pubkey) ------------------->|                                  |
     |                                     |<-- get_offer(room_id,            |
     |                                     |    invite_token) ----------------|
     |                                     |-- offer, ecdh_pubkey(A) -------->|
     |                                     |                                  |-- generate identity + ECDH keys
     |                                     |<-- submit_answer(sdp,            |
     |                                     |    ecdh_pubkey(B)) --------------|
     |<-- check_answer: sdp, ecdh_pubkey(B)|                                  |
     |                                     |                                  |
     |== ICE negotiation (STUN/TURN, поза сигнальним вузлом де можливо) =====>|
     |                                     |                                  |
     |<========= RTCDataChannel відкрито (DTLS) ==============================|
     |                                     |                                  |
     |-- ECDH(A_priv, B_pub) -> HKDF -> AES-256-GCM session key               |
     |                                                    B: ECDH(B_priv, A_pub) -> той самий ключ
     |                                     |                                  |
     |<===== зашифровані повідомлення (AES-GCM, унікальний IV) ==============>|
```

## Ключові принципи

- **Мультиакаунтність та ефемерність** — клієнт може створити перманентний зашифрований профіль (IndexedDB, Фаза 2) або одноразовий "світ" ключів, що існує лише в оперативній пам'яті вкладки і зникає без сліду при закритті.
- **Селективна інфраструктура** — адреса сигнального сервера та список STUN/TURN серверів вводяться користувачем в UI, нічого не хардкодиться в клієнті.
- **Zero-database бекенд** — сигнальний вузол не використовує СУБД; єдине сховище — самоочищувані JSON-файли з файловим блокуванням.
- **E2EE за замовчуванням** — сигнальний вузол в принципі не має технічної можливості прочитати вміст чату; максимум, що йому доступне — метадані встановлення з'єднання.

Деталі протоколу сигналінгу: [signaling-protocol.md](signaling-protocol.md). Деталі шифрування: [e2ee.md](e2ee.md). Обґрунтування рішень: [decisions.md](decisions.md).
