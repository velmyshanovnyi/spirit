# Журнал архітектурних рішень

Короткі ADR-подібні записи, щоб рішення не губились і не переобговорювались повторно.

## D1: Файлова "БД" замість СУБД для сигнального вузла

**Рішення**: сигнальний вузол зберігає стан у JSON-файлі з `LOCK_EX`, без СУБД.
**Причина**: вузол має запускатись на будь-якому shared-хостингу без прав на встановлення СУБД; координаційні дані ефемерні (TTL хвилини), тож накладні витрати СУБД невиправдані.
**Наслідок**: rate-limiting і GC мають бути реалізовані вручну на рівні файлу (див. [signaling-protocol.md](signaling-protocol.md)), а не покладатись на транзакції СУБД.

## D2: WebRTC DataChannel як транспорт повідомлень

**Рішення**: весь вміст переписки йде напряму пір-до-піра через `RTCDataChannel`, сервер ніколи не бачить контенту.
**Причина**: ключова вимога — відсутність постійної серверної інфраструктури та централізованої точки збору даних користувачів.
**Наслідок**: потрібен окремий шар сигналінгу для узгодження з'єднання (SDP/ICE), і STUN/TURN для проходження NAT — обидва обираються користувачем, не хардкодяться.

## D3: Відмова від email/телефону/OAuth

**Рішення**: єдиний ідентифікатор — локальна пара криптографічних ключів.
**Причина**: анонімність — ключова філософія проєкту; будь-яка зовнішня ідентифікація (навіть номер телефону) створює прив'язку до реального користувача.
**Наслідок**: відновлення акаунта неможливе без бекапу приватного ключа — це свідомий trade-off, задокументований для користувача в UI (Фаза 1).

## D4: Invite-токени замість хардкодженого whitelist публічних ключів

**Рішення**: доступ до кімнати сигналінгу видається одноразовим динамічним токеном, а не статичним списком ключів у конфігу PHP.
**Причина**: початковий чорновий дизайн вимагав від оператора вузла заздалегідь прописувати публічні ключі обох співрозмовників у коді — це виключає використання вузла новими/незнайомими парами користувачів і суперечить принципу анонімності та простоти (публічний вузол, яким може скористатись будь-хто, домовившись поза каналом).
**Наслідок**: статичний whitelist лишається як опційний додатковий режим для приватних вузлів закритого кола контактів, але не є дизайном за замовчуванням.

## D5: Заборона `Access-Control-Allow-Origin: *`

**Рішення**: за замовчуванням same-origin деплой; крос-origin — тільки явний whitelist origin'ів, що рефлектиться вибірково.
**Причина**: у поєднанні з можливістю створювати/читати сесії, wildcard CORS дозволяє будь-якому сайту в браузері жертви робити запити від її імені до сигнального вузла.
**Наслідок**: клієнтський HTML за замовчуванням рекомендується розміщувати на тому ж хості, що й PHP-вузол.

## D6: E2EE обов'язковий для MVP, не відкладена функція

**Рішення**: ECDH + AES-GCM шифрування кожного повідомлення в DataChannel реалізується одразу у Фазі 1, а не як майбутнє покращення.
**Причина**: сигнальний вузол за архітектурою — замінний і обирається користувачем, тобто за замовчуванням недовірений компонент, здатний побачити SDP/ICE. Без E2EE конфіденційність переписки залежала б від довіри до чужого сигнального вузла, що суперечить filozofії "нульової залежності від інфраструктури".
**Наслідок**: Фаза 1 містить криптографічний хендшейк (ECDH-обмін через сигналінг) до відкриття UI чату — трохи більше складності на старті, але без цього MVP не відповідав би заявленим гарантіям приватності.

## D7: ECDH/ECDSA на P-256 замість X25519/Ed25519

**Рішення**: для MVP використовується P-256 (NIST curve) через Web Crypto API.
**Причина**: кросбраузерна підтримка Web Crypto для X25519/Ed25519 на момент написання нерівномірна; P-256 підтримується стабільно скрізь, де є `SubtleCrypto`.
**Наслідок**: варто переглянути це рішення, коли підтримка Curve25519-сімейства в Web Crypto стане повсюдною (простіша і швидша реалізація, менше підводних каменів з валідацією точок кривої).

## D8: Мнемоніка і keyfile — одне кодування секрету, не дві системи деривації

**Рішення**: backup identity-ключа реалізується як пряме кодування вже згенерованих pkcs8-байт приватного ключа (BIP39-подібний word list для мнемоніки, passphrase-зашифрований JSON для keyfile), а не через детерміновану деривацію ключа з мнемоніки (SLIP-10 і подібні).
**Причина**: детермінована деривація EC-ключів (P-256) з seed-фрази вимагає ручної роботи з еліптичною кривою (перетворення байтів у валідний приватний скаляр), що виходить за межі можливостей `SubtleCrypto` і додає custom big-number crypto код — зайва поверхня для помилок у security-критичному коді.
**Наслідок**: мнемоніка і keyfile завжди відповідають одному й тому самому реально згенерованому ключу; підтримка обох форматів backup (за вимогою користувача мати вибір) не подвоює криптографічну складність. Деталі — [accounts.md](accounts.md).

## D9: Мультипристрій через device-linking сертифікати, без централізованого сервера

**Рішення**: підтримка кількох активних пристроїв на один акаунт реалізується через делеговані device certificates (identity-ключ підписує публічний ключ кожного пристрою), синхронізація між пристроями — eventually-consistent, через прямі P2P-сесії між пристроями користувача (той самий сигнальний+E2EE стек, що й для звичайного чату).
**Причина**: у serverless-архітектурі немає центрального реєстру "які пристрої зараз належать акаунту" і немає relay-сервера для гарантованого real-time fan-out повідомлень на всі пристрої одразу. Zero-database філософія (D1) виключає введення такого реєстру.
**Наслідок**: MVP-версія мультипристрою свідомо не гарантує миттєвої доставки на всі пристрої одночасно (доставка — активному в моменті пристрою, синхронізація історії — при наступному онлайн-контакті пристроїв між собою). Для MVP-простоти identity private key копіюється на кожен зв'язаний пристрій (компроміс безпеки, задокументований в [accounts.md](accounts.md)); строгіша делегована модель без розкриття identity-ключа — Фаза 4.

## D10: Міграція з інших месенджерів = UX-паритет, технічний імпорт — опційний і відкладений

**Рішення**: MVP не містить парсерів експортів Telegram/WhatsApp тощо. "Міграція" реалізується як простий, знайомий за термінологією onboarding, а перенесення соціального графу відбувається через надсилання invite-лінків у месенджерах, якими користувач уже користується.
**Причина**: відсутність зовнішнього ідентифікатора (D3) робить автоматичне зіставлення контактів з інших сервісів неможливим без ризику деанонімізації; пряма інтеграція з чужими API суперечила б принципу нульової залежності від сторонньої інфраструктури.
**Наслідок**: архітектура закладає точку розширення (importer-модуль з чітким інтерфейсом для контактів і історії чату), щоб додати офлайн-імпорт із файлів експорту пізніше без переробки ядра — див. [migration.md](migration.md). Реалізація цих імпортерів — опційна, поза базовим MVP.

## D11: Верифікація зовнішньої ідентичності через самостійно опубліковані URL-докази, не через центральний реєстр

**Рішення**: доказ зв'язку "цей Spirit-акаунт = той самий власник Telegram/сайту/GitHub" реалізується як самостійно опублікований, підписаний identity-ключем текстовий блок за URL, який кожен учасник (не лише Spirit) може незалежно перевірити локально. Email/DKIM-доказ свідомо винесено поза MVP через суттєво вищу технічну складність.
**Причина**: центральний "видавець галочок" суперечив би D1 (zero-database) і D3 (без центральної довіри); модель Keybase/keyoxide вже доведена й не потребує жодної Spirit-інфраструктури для самої перевірки.
**Наслідок**: один універсальний механізм покриває всі текстові/веб-платформи одразу, без окремої інтеграції під кожну; email лишається можливим майбутнім розширенням ([identity-verification.md](identity-verification.md)).

## D12: `fetch_proof` — опційна, вимкнена за замовчуванням proxy-дія на сигнальному вузлі для обходу CORS

**Рішення**: замість вимоги, щоб усі докази ідентичності хостились виключно на CORS-відкритих ресурсах (що виключило б Telegram-пости/біо), сигнальний вузол може опційно надавати stateless single-GET proxy з жорсткими SSRF/rate-limit обмеженнями.
**Причина**: більшість вебсторінок (зокрема Telegram web-прев'ю) не віддають CORS-заголовків, тож прямий `fetch()` з браузера співрозмовника не спрацює для типових платформ, де людина реально хоче опублікувати доказ.
**Наслідок**: це підвищує поверхню атаки вузла (потенційний SSRF/відкритий proxy), тому дія вимкнена за замовчуванням і оператор вмикає її свідомо, з обмеженнями, задокументованими в [signaling-protocol.md](signaling-protocol.md).

## D13: Proof-set (перелік доказів + revoke-список) доставляється напряму від власника ключа, а не зберігається на сервері

**Рішення**: список активних і явно відкликаних доказів ідентичності — підписаний identity-ключем JSON-блок, що передається контакту напряму під час P2P-хендшейку (та сама eventually-consistent модель, що й для мультипристрою, D9), а не зберігається на жодному сервері.
**Причина**: узгоджено з D1/D9 — жодна серверна інфраструктура Spirit не повинна бути єдиним джерелом правди про акаунт користувача; авторитетне джерело — сам власник identity-ключа.
**Наслідок**: явне відкликання доказу власником спрацьовує незалежно від того, чи досі існує сама публікація на зовнішній платформі; контакти бачать оновлений стан лише при наступному прямому контакті (не миттєво глобально) — прийнятний trade-off для serverless-моделі.
