# End-to-End шифрування поверх WebRTC DataChannel

## Навіщо, якщо DTLS вже шифрує канал

`RTCDataChannel` захищений DTLS — це ховає вміст від пасивного спостерігача в мережі. Але сигнальний вузол бере участь у встановленні з'єднання (бачить SDP/ICE) і за визначенням архітектури є замінним/недовіреним компонентом, обраним самим користувачем. Скомпрометований або зловмисний вузол теоретично може спробувати MITM на етапі узгодження з'єднання. Тому Spirit додає прикладний шар E2EE, незалежний від довіри до сигнального вузла. Це **обов'язкова частина MVP**, не відкладена функція.

## Ключі

Кожен клієнт при створенні акаунта (перманентного чи ефемерного "світу") генерує дві пари ключів через Web Crypto API:

1. **Identity keypair** — ECDSA P-256, для підпису/верифікації (довгостроковий, ідентифікує ноду користувача).
2. **ECDH keypair** — P-256, для узгодження сесійного ключа. Генерується заново для кожної нової P2P-сесії (ефемерність — компрометація одного сесійного ключа не розкриває минулі/майбутні сесії, forward secrecy на рівні сесій).

> P-256 обрано замість Ed25519/X25519, оскільки Web Crypto API має стабільну кросбраузерну підтримку ECDSA/ECDH на P-256 вже зараз; підтримка Ed25519/X25519 у Web Crypto ще нерівномірна між браузерами. Переоцінити цей вибір, коли підтримка X25519 стане повсюдною (див. [decisions.md](decisions.md)).

## Обмін ключами

ECDH-публічні ключі (SPKI, base64) передаються через сигнальний вузол разом із SDP offer/answer (див. [signaling-protocol.md](signaling-protocol.md), поле `ecdh_pubkey`). Публічні ключі не є секретом — їх видимість сигнальному вузлу не порушує конфіденційність переписки.

## Виведення сесійного ключа

1. Обидва клієнти обчислюють спільний секрет: `ECDH(own_private, remote_public)`.
2. Спільний секрет проганяється через HKDF (SHA-256) з фіксованим контекстним рядком (`"spirit-e2ee-v1"`) для отримання 256-бітного сесійного ключа.
3. Сесійний ключ імпортується як `AES-GCM` ключ через `SubtleCrypto.importKey`.

## Шифрування повідомлень

- Кожне повідомлення шифрується окремо: `AES-GCM` зі свіжим випадковим 96-бітним IV на кожне повідомлення (IV передається разом із шифротекстом у відкритому вигляді — це стандартна практика для GCM).
- Формат payload у DataChannel: `iv (12 байт) || ciphertext+tag`.
- Отримувач дешифрує через `SubtleCrypto.decrypt`; помилка автентифікації GCM (підробка/пошкодження) — повідомлення відкидається, з'єднання не розривається автоматично (можлива тимчасова мережева похибка).

## Захист від MITM з боку сигнального вузла

Оскільки ECDH-публічні ключі проходять через потенційно недовірений вузол, теоретично можливий MITM-атакуючий вузол, що підміняє ключі обох сторін на власні. Мітигація для MVP:

- **Safety number / fingerprint**: клієнт обчислює короткий хеш (наприклад SHA-256, скорочений до 6-8 груп цифр) від конкатенації обох identity-публічних ключів (не ECDH — довгострокових identity-ключів, підписаних власником). Обидва співрозмовники можуть звірити це число поза каналом (голосом, іншим месенджером) для впевненості, що з'єднання не перехоплене.
- Це **опційна ручна дія користувача**, не блокує звичайний анонімний чат — задокументовано як рекомендація для high-security сценаріїв, аналогічно "safety number" в Signal.
- На майбутнє (поза MVP) можна розглянути підпис ECDH-ключа identity-ключем (authenticated key exchange), щоб хоча б детектувати підміну автоматично — зафіксовано в [roadmap.md](roadmap.md) як подальше вдосконалення.

## Аудіо/відео (дзвінки)

Медіа-потоки (`getUserMedia` + `RTCPeerConnection` track-renegotiation, той самий вже-відкритий канал, що й DataChannel) захищені наскрізно на рівні самого WebRTC: **DTLS-SRTP обов'язковий у специфікації WebRTC** (не опційна фіча Spirit) — кожен медіа-пакет шифрується DTLS-узгодженим ключем між двома peer'ами напряму. Якщо з'єднання йде через TURN-релей (Секція P1 бектрогу, форсований relay-режим), релей лише пересилає вже зашифровані UDP/TCP-пакети — він не є DTLS-endpoint і не може їх розшифрувати. Тобто конфіденційність аудіо/відео не залежить від довіри до сигнального чи TURN-вузла так само, як і для DataChannel-повідомлень.

**Важливо**: ratchet-шар (`client/js/ratchet.js`, Секція P2b) наразі покриває ЛИШЕ текстові чат-повідомлення (`btn-send`), не медіа-потоки. Це усвідомлене обмеження, не прогалина: forward secrecy для медіа через per-frame ratcheting вимагав би Insertable Streams (`RTCRtpScriptTransform`/`createEncodedStreams`) — API з нерівномірною кросбраузерною підтримкою (Safari відстає від Chrome/Firefox), що конфліктувало б з архітектурним інваріантом кросплатформності (`CLAUDE.md`). Якщо колись знадобиться post-compromise security саме для медіа (а не лише для тексту), це окрема секція бектрогу, що вимагає окремого обговорення з користувачем через explicit trade-off (WebRTC-рівня forward secrecy per-сесія — уже є природно, оскільки ECDH/DTLS-ключі генеруються заново на кожну сесію; додатковий per-frame ratchet дав би forward secrecy й PCS ВСЕРЕДИНІ однієї довгої сесії дзвінка, що для Spirit наразі не пріоритет).

## Зберігання ключів

- Ефемерний "світ": усі ключі — лише в пам'яті вкладки (`CryptoKey` об'єкти, `extractable: false` де можливо), зникають при закритті вкладки.
- Перманентний профіль (Фаза 2): identity-keypair зберігається в IndexedDB, сам захищений паролем/passphrase користувача (ключ шифрування сховища виводиться з passphrase через PBKDF2/Argon2, не зберігається у відкритому вигляді).
- Сесійні AES-GCM ключі ніколи не персистяться — генеруються наново при кожному встановленні з'єднання.
