# Опціональна верифікація зовнішньої ідентичності

## Проблема

Spirit за замовчуванням анонімний (D3) — акаунт це лише пара ключів, без прив'язки до реальної особи. Але деякі користувачі хочуть протилежного: довести співрозмовнику, що конкретний Spirit-акаунт справді керується тією ж людиною, яка володіє відомим Telegram-каналом, вебсайтом, GitHub-профілем тощо ("не анонімний, а визнаний").

Це **не** верифікація особи (KYC) — Spirit не підтверджує ім'я/паспорт. Це **прив'язка публічних ідентичностей** ("linked identity proof"): доказ, що один і той самий власник контролює і цей Spirit-ключ, і той зовнішній акаунт. Модель — та сама, що в Keybase/keyoxide: самостійно опубліковане, криптографічно перевірюване твердження.

Функція повністю опціональна й не змінює базову модель акаунта (D3 лишається чинним для тих, хто не хоче її використовувати).

## Чому не централізований реєстр верифікацій

Spirit не має сервера, якому можна було б довірити роль "видавця галочок" (це суперечило б D1/D3 — zero-database, без центральної точки довіри). Тому перевірка має бути:
- **Самостійно перевірюваною** — будь-хто (не тільки Spirit-сервер) може незалежно, локально в браузері, перевірити доказ у будь-який момент.
- **Не залежною від одного оператора вузла** — навіть якщо конкретний сигнальний вузол зникне чи стане шкідливим, раніше опубліковані докази лишаються перевірюваними напряму з зовнішньої платформи.

## Механізм: універсальний URL-доказ

Один механізм покриває всі платформи (Telegram публічний пост/біо, особистий вебсайт, GitHub gist, соцмережі) — не потрібно окремої інтеграції під кожну. Email навмисно виключений з MVP: потребує окремого DKIM-механізму, суттєво складнішого технічно (розбір DKIM-підпису листа, звірка публічного DKIM-ключа домену через DNS) — залишено як можливе розширення поза MVP.

### Формат proof-блоку

Клієнт генерує текстовий блок для публікації:

```
-----BEGIN SPIRIT PROOF-----
version: 1
identity: <base64 identity public key ідентичності, що публікує доказ>
statement: I control this account and my Spirit identity fingerprint is <короткий fingerprint>
timestamp: <unix ts>
nonce: <випадкові 16 байт, hex>
signature: <base64 ECDSA-підпис over усіх полів вище>
-----END SPIRIT PROOF-----
```

Користувач публікує цей блок будь-де, де має право писати публічно доступний контент (текст поста, біо, сторінка сайту, gist) — і зберігає посилання (URL) на цю публікацію.

### Публікація (сторона того, хто хоче підтвердитись)

1. Профіль → "Підтвердження акаунтів" → "Додати підтвердження".
2. Клієнт генерує proof-блок (підписаний власним identity private key).
3. Користувач копіює блок, публікує за обраним URL (пост, біо, сторінка), вводить цей URL назад у Spirit.
4. Клієнт одразу перевіряє власну публікацію (sanity check), додає запис `{ url, added_at }` до локального **proof-set**, збільшує version, підписує proof-set заново своїм identity-ключем.

### Приклад: публікація в Telegram

Емпірично перевірено (тестова група `https://t.me/spiritid`, липень 2026):

- `https://t.me/<name>` (звичайна прев'ю-сторінка) — показує лише назву, аватар і лічильник учасників, без тексту повідомлень. **Непридатно** як URL-доказу.
- `https://t.me/s/<name>` (режим "стрічки для пошукових ботів") — працює лише для публічних каналів (broadcast); для звичайної групи текст постів так само не віддається. **Непридатно** для більшості випадків.
- **`https://t.me/<name>/<message_id>?embed=1`** — офіційний embed конкретного повідомлення (той самий, що використовує `telegram-widget.js`). Сирий HTTP-респонс (перевірено `curl`, без виконання JS) містить текст повідомлення прямо в розмітці:
  ```html
  <div class="tgme_widget_message_text js-message_text" dir="auto">спробуй...</div>
  ```
  Це **працює навіть для звичайної (не публічно проіндексованої) групи** — достатньо знати посилання на конкретне повідомлення. Саме цей формат URL слід використовувати для proof-блоку в Telegram: опублікувати блок як окремий пост, а як URL доказу вказати `https://t.me/<name>/<message_id>?embed=1`.
  - `curl -I` на цей URL не повертає заголовка `Access-Control-Allow-Origin` — отже, прямий `fetch()` з браузера співрозмовника буде заблокований CORS, і для Telegram-доказів завжди знадобиться `fetch_proof` proxy (D12), пряма спроба фетчу тут не спрацює.
  - Відповідь також містить `Set-Cookie` — `fetch_proof` на сигнальному вузлі має ігнорувати/не пересилати cookies цільового ресурсу (не зберігати сесійний стан, лишатись stateless per D12).

### Перевірка (сторона співрозмовника)

1. При хендшейку/оновленні контакту клієнт отримує від співрозмовника підписаний proof-set (див. нижче — "Доставка proof-set").
2. Для кожного запису `{url}` клієнт отримує вміст сторінки:
   - Спочатку пряма спроба `fetch(url)` — працює для ресурсів з відкритим CORS (власний сайт, GitHub raw gist тощо).
   - Якщо пряма спроба заблокована CORS — фолбек через **proxy-дію обраного сигнального вузла** (`fetch_proof`, див. [signaling-protocol.md](signaling-protocol.md)) — той самий вузол, якому користувач уже довіряє для сигналінгу, виконує один stateless GET-запит на його користь.
3. З отриманого тексту витягується блок між маркерами `BEGIN/END SPIRIT PROOF` (регулярним виразом — дозволяє доказу бути частиною більшого поста/сторінки з іншим текстом навколо).
4. Перевіряється: (а) поле `identity` в блоці збігається з identity public key контакту, з яким зараз ведеться чат; (б) підпис валідний відносно цього ж публічного ключа.
5. Якщо все ок — показується бейдж (іконка платформи/URL + "перевірено: `<дата>`"). Клікабельний, розкриває сам URL для ручної перепевірки користувачем.

## Доставка proof-set (без центрального реєстру)

Proof-set — підписаний identity-ключем список активних і відкликаних доказів, доставляється **напряму від власника ключа контакту**, тим самим каналом, що вже використовується для обміну identity/ECDH публічних ключів під час P2P-хендшейку (той самий патерн, що й device certificates, D9):

```json
{
  "version": 3,
  "proofs": [
    { "url": "https://t.me/username/123", "label": "telegram", "added_at": 1234567890 }
  ],
  "revoked": [
    { "url": "https://example.com/old-proof", "revoked_at": 1234500000 }
  ],
  "signature": "<base64 ECDSA over version+proofs+revoked>"
}
```

- Контакт кешує останній отриманий proof-set локально; оновлюється щоразу, коли обидва боку знову виходять на зв'язок (той самий eventually-consistent підхід, що й для мультипристрою, D9 — тут немає потреби в постійному сервері зберігання).
- Це навмисно узгоджено з тим, що доставку "хто зараз кому має вірити" визначає сам власник ключа, а не зовнішня платформа чи Spirit-вузол.

## Життєвий цикл бейджа: періодична переперевірка

- Клієнт-контакт періодично (наприклад, раз на добу, і за явним запитом користувача "Перевірити зараз") повторно фетчить URL і перевіряє підпис.
- Успіх → оновлюється мітка "перевірено: `<дата>`".
- Неуспіх (сторінка недоступна, блок видалено, підпис більше не збігається) — бейдж не зникає одразу (можливий тимчасовий збій мережі чи хостингу), а переходить у стан **"не вдалося підтвердити востаннє: `<дата>`"** після кількох послідовних невдалих спроб. Це відрізняється від явного відкликання (нижче) — тут причина невідома і може бути тимчасовою.

## Явне відкликання (revoke) власником

Користувач може прибрати запис зі свого proof-set навіть без видалення публікації на зовнішній платформі (наприклад, забув пароль від Telegram, чи просто передумав):

1. Профіль → "Підтвердження акаунтів" → обраний запис → "Відкликати".
2. Запис переноситься з `proofs` у `revoked` (з `revoked_at`), version proof-set збільшується, весь блок підписується заново.
3. При наступному контакті співрозмовники бачать оновлений proof-set: бейдж зникає, а якщо раніше показувався — може відображатись позначка "раніше підтверджено, відкликано власником `<дата>`" замість мовчазного зникнення (прозоріше для UX, ніж просто "було й нема").

Відкликання авторитетне з боку власника ключа незалежно від того, чи досі існує сама публікація на зовнішній платформі.

## Безпекові застереження

- Це доказ **зв'язку акаунтів**, не доказ особи. Скомпрометований identity private key ламає і чат, і всі доведені зв'язки одночасно — окрема модель загроз не потрібна понад ту, що вже описана для E2EE ([e2ee.md](e2ee.md)) і мультипристрою ([accounts.md](accounts.md)).
- `fetch_proof` proxy-дія на сигнальному вузлі — це фактично відкритий URL-fetch proxy, що додає SSRF/DoS-поверхню атаки на вузол. Тому вона: (а) вимкнена за замовчуванням, оператор вузла вмикає явно; (б) обмежена: тільки `http(s)`, заборона приватних/loopback IP-діапазонів, жорсткий timeout і ліміт розміру відповіді, той самий rate-limiting, що й інші дії вузла (див. [signaling-protocol.md](signaling-protocol.md)).
