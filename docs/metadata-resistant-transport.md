# Дослідницька нотатка: метадані-стійкий транспорт (Секція P1(б)/(в))

Частина Секції P1 ([specs/phase5/security-hardening.md](../specs/phase5/security-hardening.md)). Секція P1(а) (UI-перемикач "форсувати TURN-релей") реалізується кодом окремо — ця нотатка покриває лише (б) і (в), для яких спека explicitly вимагає почати з дослідження, не з коду.

## (б) Власний STUN/TURN на наявній інфраструктурі (kolomedi/kibr)

**Висновок: технічно неможливо на поточній інфраструктурі без зміни хостингу.**

Обидва тестові вузли (`kibr.ftp.tools`, `kolomedi.ftp.tools`, див. `deploy/hosts.local.md`, локальний файл) — це shared PHP-хостинг з доступом лише по FTP. Це означає:

- **Немає SSH/shell-доступу** — не можна встановити й запустити демон (`coturn` чи будь-який інший TURN-сервер), лише завантажити файли по FTP, які виконує вже наявний веб-сервер (Apache/nginx + PHP-FPM).
- **Немає довготривалого процесу** — TURN-сервер повинен бути постійно запущеним процесом, що слухає UDP (типово порт 3478 + діапазон relay-портів). PHP-скрипт, що виконується на HTTP-запит і завершується, структурно не може це замінити.
- **Немає контролю над мережевими портами** — навіть якби існував спосіб запустити процес (наприклад, через cron-хак чи CGI), shared-хостинг практично завжди блокує вхідні з'єднання на довільні UDP-порти ззовні (лише 80/443 гарантовано відкриті).

**Практичний висновок**: власний TURN на цій інфраструктурі означав би або (1) перехід на VPS/dedicated-хостинг з shell-доступом для хоча б одного з вузлів — архітектурна зміна, що виходить за межі "той самий деплой, що й зараз", і потребує явного рішення користувача про новий хостинг-провайдер і витрати, або (2) використання стороннього TURN-сервісу (безкоштовні/дешеві опції: [Metered.ca](https://www.metered.ca/tools/openrelay/) (безкоштовний open-relay STUN/TURN, обмежена пропускна здатність), Twilio Network Traversal Service (платний, по використанню), Cloudflare Calls TURN (у публічній беті на момент написання) — усі просто дають `urls`/`username`/`credential` для `RTCConfiguration.iceServers`, той самий формат, що вже використовує `stun-url` на екрані «Сервер»).

Варіант (2) НЕ суперечить жодному архітектурному інваріанту проєкту (D1-D12) — TURN-сервер лише ретранслює вже E2EE-зашифрований WebRTC-трафік, не бачить контенту (DTLS-SRTP), максимум бачить метадані (IP обох сторін і факт з'єднання) — той самий клас довіри, що й сигнальний вузол уже має. Якщо користувач вирішить прийняти сторонній TURN, Секція P1(а) (UI-перемикач forced-relay + поле для власного TURN URL/credentials на екрані «Сервер», за тим самим патерном, що й наявне поле `stun-url`) вже покриває це технічно — просто користувач вписує туди чужий/сторонній TURN URL замість власного. Окремого рішення чи коду для "власного" TURN не потрібно, поки немає VPS.

## (в) Опційний Tor-транспорт для сигналінгу

**Висновок: суттєва зміна протоколу, не UI-перемикач. Не рекомендовано як наступний крок без окремого явного рішення користувача про обсяг роботи.**

### Що саме потребувало б зміни

Поточний сигналінг (`specs/phase1/mvp.md`, `docs/signaling-protocol.md`) — простий HTTP-поллінг (`fetch()` до PHP-ендпоінтів `create_invite`/`create_offer`/`get_offer`/`submit_answer`/`check_answer`). Щоб сигналінг ходив через Tor:

1. **Сигнальний вузол мав би бути доступний як onion-сервіс** (`.onion`-адреса) — вимагає Tor-демона на боці сервера (та сама shell-доступу проблема, що й для TURN вище: shared PHP-хостинг не дає запустити `tor` як довготривалий процес). Альтернатива — тримати `.onion` окремо на власному VPS/Raspberry Pi, який проксує на існуючий PHP-бекенд; це вже інфраструктурний проєкт, не код у цьому репо.
2. **Клієнт (браузер) не має нативного доступу до Tor-мережі** — звичайний `fetch()` не вміє `.onion`. Реальні опції:
   - **Tor Browser / браузер із вбудованим Tor-проксі**: якщо користувач сам запускає Spirit у Tor Browser, `fetch()` до `.onion`-URL працює прозоро — жодних змін у клієнтському коді Spirit не потрібно, окрім дозволу вводити `.onion`-URL у полі "Сигнальний вузол" (тривіальна зміна, це вже текстове поле). Це найдешевший шлях і, ймовірно, достатній: **Spirit не заново винаходить Tor-транспорт, а просто не блокує `.onion`-URL, якщо користувач сам приносить Tor**.
   - **snowflake-подібний pluggable transport безпосередньо в звичайному браузері** (без Tor Browser): вимагає WebRTC-based bridge (Snowflake сам використовує WebRTC data channels для обфускації) — це вже нетривіальна нова підсистема, конфліктує з "не просто UI-перемикач" застереженням спеки. Реалізація потребувала б окремого дизайн-документа й, найімовірніше, зовнішньої залежності (vendored ESM, за інваріантом "без bundler").
3. **WebRTC P2P-з'єднання після сигналінгу все одно НЕ йде через Tor** (Tor не проксує UDP/WebRTC ICE-трафік за задумом) — навіть з `.onion`-сигналінгом, пряме P2P-з'єднання (чи навіть relay через TURN) все ще розкриває реальні IP обох сторін одна одній чи TURN-серверу, якщо не форсувати relay через TURN, який сам собі теж не проходить через Tor. Tor тут захищає **лише метадані сигналінгу** (хто з ким домовляється про з'єднання, коли), не сам факт WebRTC-з'єднання і не IP-адреси учасників одне для одного. Це важливе уточнення моделі загроз: Tor-сигналінг сам по собі не замінює forced-TURN (P1(а)) для приховування IP від співрозмовника — це два незалежні, взаємодоповнювальні заходи.

### Рекомендація

**Перевірено (2026-07-18)**: `client/js/signalingClient.js`'s `apiRequest` викликає звичайний `fetch(baseUrl, {...})` без жодного protocol/regex-фільтра на `baseUrl` — поле "Сигнальний вузол" на екрані «Сервер» вже приймає й передає будь-який URL, включно з `.onion`, без змін коду. Користувач Tor Browser вже сьогодні може вписати туди `.onion`-адресу сигнального вузла (за умови, що сам вузол доступний як onion-сервіс — інфраструктурна, не клієнтська задача) і сигналінг піде через Tor прозоро. Дій не потрібно — пункт закритий дослідженням, не кодом.

Повноцінний snowflake-подібний bridge для звичайних (не-Tor-Browser) користувачів — окрема, велика секція, що потребує явного рішення користувача про обсяг (нова зовнішня залежність, новий протокол узгодження) ПЕРЕД будь-яким кодом, per застереження спеки. Не починати без окремого AskUserQuestion.
