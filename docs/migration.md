# Онбординг та міграція з інших месенджерів

## Чому "міграція" в класичному сенсі неможлива

Telegram/WhatsApp/Signal ідентифікують користувачів через номер телефону; email-акаунти прив'язані до пошти. Spirit принципово не має жодного зовнішнього ідентифікатора (див. [decisions.md](decisions.md), D3) — тож технічно "знайти моїх контактів з Telegram у Spirit" неможливо: система в принципі не знає, хто ці люди, доки вони самі не з'являться зі своїм ключем.

Тому "міграція" в Spirit — це не перенесення акаунта, а **зниження бар'єру входу та психологічної незвичності** для людини, яка звикла до звичних месенджерів.

## Дефолтна стратегія (Фаза 1, MVP): UX-паритет без технічного імпорту

- Термінологія інтерфейсу — звична, не крипто-жаргон: "Контакти", "Запросити друга", "Посилання-запрошення" замість "identity pubkey", "invite token" в UI-тексті (технічні терміни лишаються у коді/докладних налаштуваннях).
- Onboarding — максимум 3 кроки: створити акаунт → опційний backup → надіслати перше invite-посилання.
- **Соціальний граф переноситься природним шляхом, поза Spirit**: invite-лінк на приєднання до чату користувач надсилає через месенджер, яким уже користується співрозмовник (Telegram, SMS, WhatsApp) — не через технічний імпорт, а через звичайний "ось лінк, перейди". Це не вимагає доступу до чужих акаунтів чи API сторонніх сервісів.
- Наслідок: у MVP немає жодного парсера чужих експортів — і не потрібен, щоб отримати зручний онбординг.

## Архітектурні хуки для майбутнього технічного імпорту (опційно, поза MVP)

Хоча в MVP імпорт не реалізується, архітектура закладає точку розширення — окремий **importer-модуль** з чітким інтерфейсом, щоб додати підтримку пізніше без переробки ядра.

### Імпорт контактів (опційний, Фаза 2b)

Інтерфейс:
```ts
parseContactList(file: File): { displayName: string; sourceIdentifier: string }[]
```
Реалізації під конкретні формати підключаються окремо: Telegram contacts export (JSON), vCard (.vcf), WhatsApp contacts export.

- Імпортований контакт зберігається як **"очікує на верифікацію"** — заглушка в списку контактів без Spirit identity pubkey.
- Активується лише коли ця людина сама з'явиться в Spirit (через invite) — **зіставлення тільки за явною згодою користувача** (ручне підтвердження "це той самий Іван" по displayName), без автоматичного зливання будь-яких зовнішніх ідентифікаторів у профіль контакту. Це свідомо повільніше за автоматичний match, щоб не створювати механізм деанонімізації через збіг імен/номерів.

### Імпорт історії переписки (опційний, Фаза 2b)

Інтерфейс:
```ts
parseChatExport(file: File, format: "telegram-json" | "whatsapp-txt"): { timestamp: number; sender: string; text: string }[]
```
- Імпортовані повідомлення позначаються міткою **"історичне (імпортоване)"** в UI — вони не проходили через Spirit P2P/E2EE, це локальна копія тексту, без криптографічної гарантії цілісності чи автентичності відправника.
- Прив'язуються до контакту через те саме ручне зіставлення, що й імпорт контактів.
- Зберігаються в тому самому зашифрованому IndexedDB-профілі, що й нативна історія Spirit (Фаза 2).

## Чому не пряма API-інтеграція з іншими месенджерами

Пряма інтеграція (читання чатів Telegram/WhatsApp напряму) вимагала б облікових даних або токенів доступу користувача до сторонніх сервісів — це суперечить анонімності Spirit і принципу нульової залежності від чужої інфраструктури. Тому будь-який імпорт — виключно offline, з файлів експорту, які користувач сам ініціює та завантажує локально.
