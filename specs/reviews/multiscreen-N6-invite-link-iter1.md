## Знахідки

Немає. Перевірено п'ять напрямів:

1. **Побудова посилання** — `new URL(pathname, origin)` + `.search`/`.hash` коректна; `encodeURIComponent` при побудові й симетричне декодування `URLSearchParams.get` при отриманні; порт origin зберігається.
2. **Дефолт `locationSearch`** — `doc` уже прив'язаний до моменту обчислення дефолту (JS обчислює параметри зліва направо); `doc.defaultView` ніде не гарантовано null у продакшн-шляху (і вже розіменовується без перевірки в інших місцях коду).
3. **Чотири точки навігації** прослідковано: btn-generate (безумовно "room" — коректно), unlock/backup-skip (`postIdentityRoute()`), device-link-grant (безумовно "profile" — коректно виключено). `cameFromInviteLink` обчислюється один раз, ніколи не переоцінюється — без подвійної навігації.
4. **Витік токена через Referer** — за замовчуванням `strict-origin-when-cross-origin` (немає `<meta name="referrer">` override), query-рядок обрізається з Referer на крос-origin запитах (Google GSI); токен — лише рандеву-креденшел, не криптографічна основа (E2EE тримається на ECDH+TOFU fingerprint).
5. **Тести** — невакуумні (clipboard-тест перевіряє точний вміст виклику, missing-fields — порожній display, regression-тест підтверджує незмінність звичайної сесії); DI-параметр `locationSearch` не послаблює продакшн-гарантію (продакшн-виклик `initApp(document)` без опцій завжди використовує реальний дефолт).

Переклади вибірково перевірені (de/es/fr/it/ru) — точні, узгоджені з англійськими ключами.

Живо перевірено в реальному браузері (не jsdom, через прямі DOM-виклики `.click()` — `preview_click`-тул виявився ненадійним цю сесію): повний round-trip — створення ефемерного акаунта → заповнення room-id/token → копіювання → відкриття того самого посилання (справжній reload із query-рядком) → префіл полів → створення профілю → backup-крок коректно НЕ переходить завчасно → «Пропустити» → авто-перехід на «Кімната» зі збереженими полями.
