## Знахідки та рішення

1. **[HIGH, ВИПРАВЛЕНО]** `ownProofSetCache`/`lastGeneratedProofBlockText` (обидва — стан рівня вкладки в `app.js`, Секція E) НІКОЛИ не скидались при перемиканні активного профілю в межах ОДНІЄЇ вкладки (`btn-profile-unlock`, `btn-profile-confirm`, `btn-generate`, приєднання нового пристрою — усі мутують `state.identityKeyPair`/`state.senderKey` без повторного виклику `initApp`). Конкретний сценарій: розблокувати профіль A → згенерувати/додати доказ (кеш заповнюється даними A) → розблокувати профіль B у ТІЙ САМІЙ вкладці → UI усе ще показує докази A під ідентичністю B; гірше — дія відкликання/додавання перепідписала б застарілий набір A ключем B і зберегла його під ключем сховища B, пошкоджуючи дані. Виправлено: нова функція `resetOwnProofsState()` викликається одразу після кожної з чотирьох точок зміни активного профілю. Покрито новим regression-тестом "resets own-proof state ... when a different profile becomes active in the same tab", який точно відтворює сценарій (RED до фіксу, GREEN після).
2. **[LOW/MED, ВИПРАВЛЕНО]** Прямий `fetch(url)` у `fetchProof.js` виконувався для БУДЬ-ЯКОГО URL з proof-set контакту (повністю недовірений вхід від співрозмовника) без перевірки схеми — клієнт міг спробувати `fetch("file:///...")`/`javascript:`-подібні URL. Виправлено: `isFetchableDirectly()` дозволяє прямий fetch лише для `http:`/`https:`, інакше одразу йде через серверний `fetch_proof` (який і так має власний SSRF-захист/allowlist на бекенді). Не усуває сам факт, що `https://`-URL від недовіреного контакту може бути маяком для збору IP/статусу онлайн (браузерна модель безпеки й так блокує читання cross-origin тіла відповіді та не передає credentials) — задокументовано як прийнятний залишковий ризик, не блокуючий.
3. **[LOW, ВИПРАВЛЕНО якість тесту]** Тест дедуплікації `setInterval` раніше стверджував слабку нерівність (`callsAfterTwoDays - callsAfterOneDay <= callsAfterOneDay`), яка НЕ довела б регресію подвоєння інтервалу. Замінено на прямий підрахунок викликів `setInterval`/`clearInterval` через spy — тепер тест справді ловить регресію (перевірено вручну: тимчасове видалення `clearInterval`-виклику в коді ламало новий тест).

## Прийнято без змін

- `verifyProofBlock`-заснований sanity-check у `btn-add-proof` коректний і не обходиться: вимагає одночасно збігу `identity`-поля з очікуваним І валідного підпису.
- `proofVerification` Map зростає необмежено (застарілі записи після видалення контакту/URL не чистяться) — лише в пам'яті, обмежений реалістичною кількістю контактів×доказів; не блокуюче, залишено без змін.
- Логіка лічильника послідовних невдач (`consecutiveFailures`) коректна: скидається до 0 при успіху, інкрементується при невдачі, коректно прив'язана до пари (fingerprint, url), без витоку між різними URL чи контактами.
- Дедуплікація `setInterval` сама по собі вже була коректною (`clearInterval` перед новим `setInterval`) — знахідка стосувалась лише слабкості ТЕСТУ, не реалізації.
