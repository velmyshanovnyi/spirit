## Контекст

Під час імплементації Секції 3 виявлено реальну архітектурну прогалину: відновлення identity з backup (лише сирі приватні байти, за D8) не мало надійного способу відновити відповідний публічний ключ — Web Crypto не має операції "derive public from private". Виправлено через JWK: експорт приватного EC-ключа за специфікацією (RFC 7518 §6.2.2) ЗОБОВ'ЯЗАНИЙ містити публічні координати `x`/`y` поряд із приватним скаляром `d`, незалежно від формату, з якого ключ імпортовано. Підтверджено емпірично в Node.

## Знахідки

Жодних блокуючих. Основна теза (гарантія JWK) підтверджена коректною й крос-рушійною (RFC-вимога, не деталь конкретної реалізації); при відсутності x/y на неконформному рушії `importKey` впаде з чіткою `DataError`, а не мовчки згенерує неправильний ключ.

Три дрібниці виправлено:
1. Тест "throws a clear error if the private key isn't extractable" використовував голий `rejects.toThrow()` без matcher — міг пройти vacuous на будь-якій сторонній помилці. Виправлено: `.rejects.toThrow(/extractable/i)`.
2. Дефолт `keyUsages = ["verify"]` був специфічний для ECDSA — виклик для ECDH без явного `[]` впав би (безпечно, але незручно). Виправлено: дефолт виводиться з `algorithm.name`.
3. Невикористана деструктуризована змінна `d` перейменована на `_privateScalar` (косметика, без зміни поведінки).

**Архітектурне розміщення**: підтверджено доречним лишити функцію в `identity.js` (не виносити в `profile.js`) — модуль уже володіє всією логікою конвертації форматів ключів, функція алгоритм-агностична (працює і для ECDSA, і для ECDH), і додавання суто адитивне — жодна вже конвергована функція Фази 1 Секції 1 не змінена.
