# Спека: Передача файлів через DataChannel (Фаза 4)

Рішення узгоджені з користувачем 2026-07-18 через AskUserQuestion: **явне підтвердження перед прийомом** (той самий патерн, що й invite-запрошення — не автозбереження), **без жорсткого ліміту розміру файлу** (лише попередження в UI, оскільки весь файл тримається в RAM під час передачі — zero-database означає жодного тимчасового серверного буфера).

## Контекст

Наявний чат уже має повний P2P E2EE DataChannel (`state.channel`, `client/js/app.js`) з ratchet-шифруванням user-typed тексту (Секція P2b) і статичним `sessionKey`-шифруванням control-повідомлень (`device-list-announce`, `push-subscription-announce` тощо, `encryptMessage`/`decryptMessage`, `client/js/e2ee.js`). Усі control-повідомлення сьогодні — JSON, серіалізований у текст, надісланий одним `channel.send(...)` викликом без чанкування.

## Архітектурні рішення

1. **Шифрування**: файли йдуть на статичному `sessionKey` (той самий шлях, що й control-повідомлення), НЕ через ratchet-ланцюжок user-тексту. Узгоджується з прецедентом Секції P2b (`security-hardening.md`), де ratchet свідомо звужений лише до user-typed чат-тексту — розширення на файли додало б суттєвий обсяг тестів за ту саму логіку, що вже прийнято як trade-off для control-подібних потоків.
2. **Чанкування**: DataChannel має практичний ліміт розміру повідомлення (~256KB безпечно в більшості браузерів, SCTP-фрагментація непередбачувана понад це). Файл розбивається на чанки по 16KB (сирі байти → base64 у JSON control-повідомленні `file-chunk`), консистентно з наявним патерном "усе control — JSON текст".
3. **Backpressure**: перед кожним наступним чанком перевіряється `channel.bufferedAmount` — якщо перевищує поріг (`BUFFERED_AMOUNT_HIGH_THRESHOLD`, рекомендовано 1MB), очікується подія `bufferedamountlow` (поріг `bufferedAmountLowThreshold`) перед продовженням, щоб не переповнити sending buffer WebRTC і не спричинити розрив з'єднання на великих файлах.
4. **Цілісність**: відправник рахує SHA-256 усього файлу ДО розбиття на чанки, включає в `file-offer`; отримувач перераховує після повного складання й порівнює — розбіжність = явна помилка передачі, файл не пропонується як завершений.
5. **Прийом**: отримувач бачить `file-offer` (ім'я, розмір, MIME-тип, sha256) як банер із кнопками "Прийняти"/"Відхилити" — той самий UX-патерн, що вже є для інших підтверджень користувача в проєкті. До прийняття чанки НЕ надсилаються (відправник чекає на `file-accept`/`file-reject` control-повідомлення від отримувача перш ніж почати передачу — уникає марної витрати трафіку на відхилений файл).
6. **UI-попередження про розмір**: немає жорсткого ліміту коду, але при виборі файлу понад орієнтовний поріг (рекомендовано 100MB) — попереджувальний текст про ризик використання пам'яті вкладки, без блокування дії.

## Секція FT1: Чисте ядро чанкування/збирання (без інтеграції в чат)

Той самий підхід, що вже виправдав себе для ratchet.js/shamir.js/webPushCrypto.js: спочатку самодостатній, повністю протестований модуль, окремий exec review, лише ПОТІМ — інтеграція в `app.js`.

- [ ] **Tests**: `client/tests/fileTransfer.test.js` — `splitFileIntoChunks(arrayBuffer, chunkSize)` повертає точні за розміром чанки (останній може бути коротшим), round-trip зі збиранням дає БАЙТ-В-БАЙТ ідентичний оригінал; `chunksToBase64`/`base64ToChunk` кодування без втрат; `computeFileHash(arrayBuffer)` (SHA-256 через Web Crypto) детермінований і відрізняється для різних вхідних даних; `FileAssembler`-подібний клас/функція приймає чанки в БУДЬ-ЯКОМУ порядку (мережа не гарантує порядок для JSON control-повідомлень, хоча DataChannel за замовчуванням ordered — але явна перевірка стійкості до неврегульованого порядку дешевша за припущення) і виявляє відсутні/дублікати чанків.

## Секція FT2: Control-повідомлення й інтеграція в чат-потік

- [ ] **Tests**: `client/tests/app.test.js` (доповнення) — вибір файлу в UI надсилає `file-offer` (не сирі чанки) отримувачу; отримувач бачить банер прийняття з іменем/розміром; клік "Прийняти" надсилає `file-accept`, після чого відправник починає слати `file-chunk`; клік "Відхилити" надсилає `file-reject`, чанки НІКОЛИ не надсилаються; прогрес-бар оновлюється на кожен отриманий/надісланий чанк; після останнього чанка — перевірка SHA-256, показ файлу як завершеного (з посиланням на завантаження через `Blob`/`URL.createObjectURL`) або явна помилка при розбіжності хешу; backpressure-логіка не надсилає новий чанк, поки `bufferedAmount` вище порогу (мокати `RTCDataChannel`-подібний фейк із контрольованим `bufferedAmount`/подією `bufferedamountlow`).

## Верифікація

FT1 (чисте ядро) — безпечно почати без додаткових питань, той самий патерн crypto/logic-core-first. FT2 (жива інтеграція в реальний DataChannel) потребує живої перевірки реальної передачі файлу між двома вкладками/пристроями в Browser pane чи вручну — юніт-тести з фейковим каналом не ловлять реальну поведінку `bufferedAmount`/фрагментації WebRTC (той самий клас ризику, що вже двічі підтвердився цієї сесії: Service Worker у PN3, PoW-продуктивність у SR2 — Node/vitest і review не бачать реальної браузерної поведінки).
