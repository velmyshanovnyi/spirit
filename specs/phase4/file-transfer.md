# Спека: Передача файлів через DataChannel (Фаза 4)

Рішення узгоджені з користувачем 2026-07-18 через AskUserQuestion: **явне підтвердження перед прийомом** (той самий патерн, що й invite-запрошення — не автозбереження), **без жорсткого ліміту розміру файлу** (лише попередження в UI, оскільки весь файл тримається в RAM під час передачі — zero-database означає жодного тимчасового серверного буфера).

## Контекст

Наявний чат уже має повний P2P E2EE DataChannel (`state.channel`, `client/js/app.js`) з ratchet-шифруванням user-typed тексту (Секція P2b) і статичним `sessionKey`-шифруванням control-повідомлень (`device-list-announce`, `push-subscription-announce` тощо, `encryptMessage`/`decryptMessage`, `client/js/e2ee.js`). Усі control-повідомлення сьогодні — JSON, серіалізований у текст, надісланий одним `channel.send(...)` викликом без чанкування.

## Архітектурні рішення

1. **Шифрування**: файли йдуть на статичному `sessionKey` (той самий шлях, що й control-повідомлення), НЕ через ratchet-ланцюжок user-тексту. Узгоджується з прецедентом Секції P2b (`security-hardening.md`), де ratchet свідомо звужений лише до user-typed чат-тексту — розширення на файли додало б суттєвий обсяг тестів за ту саму логіку, що вже прийнято як trade-off для control-подібних потоків.
2. **Чанкування**: DataChannel має практичний ліміт розміру повідомлення (~256KB безпечно в більшості браузерів, SCTP-фрагментація непередбачувана понад це). Файл розбивається на чанки по 16KB (сирі байти → base64 у JSON control-повідомленні `file-chunk`), консистентно з наявним патерном "усе control — JSON текст".
3. **Backpressure**: перед кожним наступним чанком перевіряється `channel.bufferedAmount` — якщо перевищує поріг (`BUFFERED_AMOUNT_HIGH_THRESHOLD`, рекомендовано 1MB), очікується подія `bufferedamountlow` (поріг `bufferedAmountLowThreshold`) перед продовженням, щоб не переповнити sending buffer WebRTC і не спричинити розрив з'єднання на великих файлах.
4. **Цілісність**: відправник рахує SHA-256 усього файлу ДО розбиття на чанки, включає в `file-offer`; отримувач перераховує після повного складання й порівнює — розбіжність = явна помилка передачі, файл не пропонується як завершений.
5. **Прийом**: отримувач бачить `file-offer` (ім'я, розмір, MIME-тип, sha256) як банер із кнопками "Прийняти"/"Відхилити" — той самий UX-патерн, що вже є для інших підтверджень користувача в проєкті. До прийняття чанки НЕ надсилаються (відправник чекає на `file-accept`/`file-reject` control-повідомлення від отримувача перш ніж почати передачу — уникає марної витрати трафіку на відхилений файл).
6. **UI-попередження про розмір**: немає жорсткого ліміту коду, але при виборі файлу понад орієнтовний поріг (рекомендовано 100MB) — попереджувальний текст про ризик використання пам'яті вкладки, без блокування дії.

## Секція FT1: Чисте ядро чанкування/збирання (без інтеграції в чат)

Той самий підхід, що вже виправдав себе для ratchet.js/shamir.js/webPushCrypto.js: спочатку самодостатній, повністю протестований модуль, окремий exec review, лише ПОТІМ — інтеграція в `app.js`.

- [x] **Tests**: `client/tests/fileTransfer.test.js` (29 tests) — `splitFileIntoChunks(arrayBuffer, chunkSize)` точні за розміром чанки (останній може бути коротшим, exact-multiple без спурного порожнього хвостового чанка, порожній файл -> 0 чанків); `chunkToBase64`/`base64ToChunk` без втрат для довільних бінарних даних (0x00, 0xff, повний діапазон байтів, 20 випадкових прогонів); `computeFileHash(arrayBuffer)` (SHA-256 hex через Web Crypto) детермінований, відрізняється для різних входів, звірений із незалежними тестовими векторами SHA-256("") і SHA-256("abc"); `createFileAssembler` приймає чанки в БУДЬ-ЯКОМУ порядку (reverse, shuffled), `isComplete`/`missingIndices` коректні на частковому стані, дублікат індексу не ламає `assemble` (first-write-wins, задокументовано), `assemble()` кидає виняток до завершення; повний round-trip split->base64->reassemble->hash на ~800KB/51 чанці при рекомендованому 16KB chunkSize.
- [x] **Impl**: `client/js/fileTransfer.js` (чисті функції, без залежності від `app.js`/DOM). Бінарно-безпечне base64-кодування через `String.fromCharCode`-по-байту + `btoa`/`atob`, консистентно з `webPushCrypto.js`'s `bytesToBase64Url`/`base64UrlToBytes` (НЕ `TextEncoder`/`TextDecoder`, які б спотворили довільні бінарні дані). Формат хешу — lowercase hex, консистентно з `identity.js`'s `fingerprint()`. `createFileAssembler`: first-write-wins на дублікат індексу.
- [x] **Exec review**: `specs/reviews/file-transfer-FT1-iter1.md` — 0 блокуючих знахідок (Opus subagent); підтверджено бінарну безпеку base64 round-trip, консистентність формату хешу, коректність duplicate-handling і off-by-one, відсутність O(n^2). Збіжність на ітерації 1; додано 4 тести на throw-guards/ArrayBuffer-гілку за інформаційною рекомендацією рев'ювера (без повторного рев'ю, лише додаткове покриття вже схваленої логіки). 629/629 тестів зелені.

## Секція FT2: Control-повідомлення й інтеграція в чат-потік

- [x] **Tests**: `client/tests/app.test.js` (доповнення, 12 нових тестів) — вибір файлу в UI надсилає `file-offer` (не сирі чанки) отримувачу; отримувач бачить банер прийняття з іменем/розміром; неверифікований peer не отримує банер (той самий гейт, що й для тексту чату); клік "Прийняти" надсилає `file-accept`, після чого відправник починає слати `file-chunk`; клік "Відхилити" надсилає `file-reject`, чанки НІКОЛИ не надсилаються; прогрес-бар оновлюється на кожен надісланий чанк; після останнього чанка — перевірка SHA-256, показ файлу як завершеного (з посиланням на завантаження через `Blob`/`URL.createObjectURL`) або явна помилка при розбіжності хешу без посилання на завантаження; `file-chunk` для невідомого/неприйнятого `fileId` тихо ігнорується; backpressure-логіка не надсилає новий чанк, поки `bufferedAmount` вище порогу, і відновлюється рівно на подію `bufferedamountlow` (мокано `RTCDataChannel`-подібний фейк із контрольованим `bufferedAmount`/подією). 640/640 тестів зелені (629 базових + 11 нових приймальних, один тест розбито на два через повторний `initApp` на тому самому document).
- [x] **Impl**: `client/js/app.js` (`CONTROL_MESSAGE_TYPES` += `file-offer`/`file-accept`/`file-reject`/`file-chunk`; гілки в `handleChatMessage` з гейтом `state.peerFingerprint`, ідентичним звичайному тексту чату; `state.outgoingFileTransfers`/`incomingFileTransfers`/`pendingFileOffers`; `sendFileChunks`/`waitForBufferedAmountLow` — реальна backpressure-логіка через `channel.bufferedAmount`/`onbufferedamountlow`; скидання файлового стану додано до `hideSafetyNumberHint()`, яка вже викликається на всіх 4 місцях скидання `peerFingerprint`), `client/index.html` (кнопка вибору файлу + прихований `<input type=file>`, банер прийняття/відхилення, контейнер прогресу/посилань на завантаження на екрані розмови), `client/js/i18n.js` (11 нових ключів `fileTransfer.*` у всіх 11 локалях, включно з попередженням про розмір >100MB).
- [x] **Exec review**: `specs/reviews/file-transfer-FT2-iter1.md` — 0 блокуючих знахідок (Opus subagent, перевірено окремо кожен з 8 запитаних пунктів: accept-гейт у коді, реальність backpressure, недосяжність download-посилання при розбіжності хешу, гейт неверифікованого peer, захист від fileId-спуфінгу, пам'ять, коректність скидання стану, XSS). Збіжність на ітерації 1. Два нефатальні (non-blocking) висновки дисміснуто з конкретним обґрунтуванням (не "здається, ок"): (N1) банер прийняття підтримує лише ОДНУ вхідну пропозицію одночасно — другий одночасний `file-offer` перезаписує `dataset.fileId` банера, перша пропозиція лишається недосяжною для UI до скидання сесії; (N2) `channel.onbufferedamountlow` — один спільний обробник на канал, тож ДВІ одночасні вихідні передачі, що обидві впираються в backpressure, можуть залишити першу "завислою" назавжди. Обидва — вузькі граничні випадки одночасних кількох передач, поза межами цієї секції (один файл за раз — основний потік); задокументовано тут як відомі обмеження першої ітерації, а не виправлено кодом.

## Верифікація

FT1 (чисте ядро) — безпечно почати без додаткових питань, той самий патерн crypto/logic-core-first. FT2 (жива інтеграція в реальний DataChannel) потребує живої перевірки реальної передачі файлу між двома вкладками/пристроями в Browser pane чи вручну — юніт-тести з фейковим каналом не ловлять реальну поведінку `bufferedAmount`/фрагментації WebRTC (той самий клас ризику, що вже двічі підтвердився цієї сесії: Service Worker у PN3, PoW-продуктивність у SR2 — Node/vitest і review не бачать реальної браузерної поведінки).
