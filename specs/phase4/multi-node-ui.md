# Спека: UI для кількох сигнальних/TURN вузлів (Фаза 4)

Найменший обсяг серед пунктів Фази 4 — чисто конфігураційна робота, без зміни протоколу чи архітектури. Поля `server-url`/`stun-url`/`force-turn-relay` (Секція P1(а), Фаза 5) сьогодні НЕ персистуються взагалі — звичайні HTML `value`-атрибути, скидаються при кожному перезавантаженні сторінки.

## Дизайн

- **Сховище**: `localStorage` (той самий рівень, що й `spirit.theme`/`spirit.locale` — не IndexedDB/`profile`-стор, бо це налаштування БРАУЗЕРА/пристрою, не профілю акаунта — той самий вузол типово використовується незалежно від того, який Spirit-акаунт зараз активний). Ключ `spirit.signalingNodes`, JSON-масив записів `{ id, name, serverUrl, stunUrl, forceTurnRelay }`.
- **UI на екрані «Сервер»**: список збережених вузлів (назва + скорочений URL), кнопка "Зберегти поточні налаштування як..." (запит імені), клік на запис у списку заповнює `server-url`/`stun-url`/`force-turn-relay` полями цього вузла (НЕ авто-перепідключає активну сесію — узгоджується з наявною ручною філософією цих полів), кнопка видалення на кожному записі.
- **Без вузлів у списку**: поведінка ідентична сьогоднішній (дефолтні значення з HTML, як і зараз) — жодної зміни для користувачів, що не використовують цю фічу.
- **Не є новим протоколом**: просто UI+localStorage над вже наявними полями, жодних змін до `signalingClient.js`/`server/`.

## Верифікація

Безпечно почати без додаткових питань — чиста UI/localStorage-фіча без нової крипто чи протоколу. Жива перевірка: збереження/перемикання/видалення вузла в реальному браузері на живому хості.
