# Спека: Соціальне відновлення акаунта через Shamir secret sharing (Секція P4)

Розширення бектрогу `specs/phase5/security-hardening.md`'s Секції P4 ("Соціальне відновлення портативного акаунта"). Рішення узгоджені з користувачем 2026-07-18 через AskUserQuestion.

## Контекст: що саме втрачається без цієї функції

Дослідження перед дизайном (2026-07-18) показало: у поточній Argon2id-схемі (Фаза 3, `specs/phase3/deterministic-accounts.md`, `client/js/deterministicIdentity.js`) втрата портативного пароля НЕ знищує зашифрований блоб — вона знищує єдиний спосіб детерміновано відтворити 32-байтний скаляр identity private key на пристрої без локальної копії профілю. Той самий 32-байтний скаляр — це вже той секрет, що кодується в мнемоніці/keyfile (`docs/accounts.md`). Отже: втрата і мнемоніки/keyfile, і локального пристрою з профілем в IndexedDB = незворотна втрата ідентичності, незалежно від того, портативний акаунт чи ні.

## Рішення (узгоджено з користувачем)

1. **Що розщеплюємо**: сам 32-байтний скаляр identity private key (`client/js/deterministicIdentity.js`'s вихід перед `importPrivateKeyFromScalar`) — той самий секрет, що вже кодується мнемонікою/keyfile. Відновлює доступ незалежно від того, чи акаунт портативний, чи звичайний перманентний профіль без збереженого backup.
2. **Як частки (shares) потрапляють до довірених контактів**: ОБИДВА канали підтримуються паралельно (той самий згенерований набір часток, два способи його роздати):
   - **Автоматичний P2P-канал**: під час налаштування відновлення користувач обирає N вже верифікованих контактів зі свого списку; кожен обраний контакт отримує свою частку зашифрованою через наявний E2EE-канал (той самий патерн, що й `device-list-announce`/`push-subscription-announce`) і зберігає її у власній IndexedDB, доки власник акаунта (чи сам довірений контакт) не видалить.
   - **Ручний QR-код/текстовий експорт**: користувач також може згенерувати ту саму частку як QR-код чи текстовий рядок і передати довіреній особі будь-яким іншим каналом (голосом, особисто, іншим месенджером) — жодна частка тоді не проходить через P2P-канал Spirit і не зберігається автоматично на нічиєму пристрої.

## Нова властивість моделі довіри (явно визнана, не прихована)

Досі жоден чужий пристрій ніколи не зберігав фрагмент чужого секрету — контакти зберігали лише публічну інформацію (fingerprint, device list, proof set, push-підписку). Ця секція вперше вводить: **фрагмент чужого приватного ключа, що зберігається на пристрої довіреного контакту**. Це свідомий, пороговий (k-з-n) компроміс: одна окрема частка нижче порогу k криптографічно марна (інформаційно-теоретична гарантія Shamir secret sharing — не потребує додаткового шифрування частки самої по собі, хоча канал доставки і сховище на боці контакту все одно варто захищати як звичайні секрети профілю). Компрометація одного (< k) довіреного контакту НЕ розкриває identity-ключ власника.

## Секція S1: Shamir secret sharing — чисте крипто-ядро (GF(256), без інтеграції)

Той самий підхід, що вже виправдав себе для `ratchet.js` (P2a) і `webPushCrypto.js` (PN1): спочатку самодостатній, повністю протестований крипто-модуль без залежності від решти застосунку, окремий exec review, і лише ПОТІМ (наступні секції) — розподіл часток і UI відновлення.

Без bundler-інваріант: реалізація на чистому Web Crypto API де можливо (`crypto.getRandomValues` для випадкових коефіцієнтів полінома), і власна арифметика GF(2^8) (той самий скінченний field, що й AES S-box/Reed-Solomon QR-кодів) — не потребує зовнішньої бібліотеки, бо GF(256) арифметика тривіальна (додавання = XOR, множення через log/antilog-таблиці).

- [x] **Tests**: `client/tests/shamir.test.js` (19 тестів) — GF(256)-арифметика (`gfAdd`=XOR, `gfMul` проти відомих AES/Reed-Solomon референсних значень, комутативність, вичерпна перевірка `gfInverse` для всіх 255 ненульових елементів); round-trip для рівно `threshold` часток і для БУДЬ-ЯКИХ підмножин `threshold` з `shares` (не лише перших); менше ніж `threshold` часток дає хибні байти без кидання винятку; `combineShares` не залежить від порядку часток; реалістичний 32-байтний секрет (розмір identity-скаляра); свіжість (два виклики `splitSecret` на тому самому секреті дають різні частки, обидва відновлюються); x=0 ніколи не використовується як частка; всі x-координати часток різні; межові випадки (`threshold > shares`, `threshold < 2`, `shares > 255`, порожній секрет, порожній вхід `combineShares`) кидають помилку.
- [x] **Impl**: `client/js/shamir.js` — GF(256) log/antilog-таблиці на генераторі 0x03 (0x02 НЕ примітивний елемент під 0x11B, порядок 51, а не 255 — виявлено й виправлено тестами до review), поліном ступеня `threshold-1` на кожен байт секрету з випадковими коефіцієнтами через `crypto.getRandomValues`, вільний член = байт секрету, x-координати часток = `1..shares` (детерміновано унікальні, ніколи 0); Lagrange-інтерполяція в GF(256) для відновлення. Нуль інтеграції з рештою застосунку (як і заплановано для S1).
- [x] **Exec review**: iter1 — [reviews/social-recovery-S1-iter1.md](../reviews/social-recovery-S1-iter1.md). Нуль знахідок, що потребують змін коду: коректність GF(256) (включно з виявленням/виправленням generator-2-регресії до review), CSPRNG на кожен коефіцієнт, унікальність x-координат і заборона x=0, відсутність витоку інформації при недостатній кількості часток, коректність Lagrange-інтерполяції й усіх межових випадків — усі підтверджені незалежно.

## Секція S2: Розподіл часток довіреним контактам (P2P-announce + QR/текстовий експорт)

Рішення узгоджені з користувачем 2026-07-18 (AskUserQuestion):

- **Сховище отриманої частки**: окремий IndexedDB object store `trustedShares` (НЕ поле в записі `contacts` — частка це дані ВІД власника акаунта, збережені НА пристрої контакту, концептуально відмінні від даних ПРО контакт). Запис: `{ ownerFingerprint, x, y (Uint8Array), threshold, totalShares, receivedAt }` — `ownerFingerprint` ідентифікує, чий це секрет (для відображення в UI "Ви зберігаєте частку відновлення для: ...", і для видалення при відкликанні).
- **UX вибору N/k**: вільний вибір з розумним дефолтом. Користувач обирає підмножину зі своїх верифікованих контактів (мінімум 2, інакше немає сенсу в порозі) — кожен обраний контакт стає одним з N. Поріг k пропонується за замовчуванням як "більшість" (`Math.ceil((N + 1) / 2)`, тобто 2-з-3, 3-з-5 і т.д.), з можливістю змінити через повзунок/select у діапазоні `[2, N]`.

- [x] **Tests**: `client/tests/trustedShares.test.js` (CRUD + overwrite-on-re-announce), `client/tests/recoveryShare.test.js` (`buildRecoveryShareAnnounce`/`parseRecoveryShareAnnounce` round-trip + exhaustive malformed-input rejection, `encodeShareAsText`/`decodeShareFromText` round-trip), `client/tests/app.test.js` additions (recovery-share-announce receive routing: saves on verified profile-mode peer, ignores malformed/pre-identity/ephemeral-mode, mirroring the existing `push-subscription-announce` tests and mocking `trustedShares.js`'s `saveTrustedShare`). 574 new assertions total on top of the 551 baseline; 575/575 green after the exec-review fixes below.
- [x] **Impl**: `client/js/trustedShares.js` (new `trustedShares` IndexedDB store, `db.js` bumped to `DB_VERSION` 2, idempotent upgrade), `client/js/recoveryShare.js` (new `recovery-share-announce` control-message builder/parser mirroring `push-subscription-announce`, plus the manual text-export/decode pair), `client/js/app.js` (receive-side routing with the same profile+verified-peer gate as `device-list-announce`; **KEY DESIGN DECISION** — an outbound pending-announce queue, since a "set up recovery with N contacts" feature has no persistent broadcast to reach contacts who aren't the CURRENT live peer: `recoveryShareOutboxKey`/`queueRecoveryShareForContact`/`drainRecoveryShareOutboxForPeer`, persisted in the `profile` IndexedDB store keyed per local profile, drained the moment any queued contact's identity-announce is next verified — the same "piggyback on connection" pattern every other announce already uses, just keyed per-recipient instead of "send to whoever is there now"), new Profile-screen `#recovery-card` UI (contact checkboxes, threshold `<select>` defaulting to `Math.ceil((N+1)/2)`, passphrase re-entry, text-export display), i18n keys across all locales.
  - **Scalar availability, resolved (not a limitation)**: a logged-in permanent profile's `state.identityKeyPair.privateKey` is deliberately non-extractable (`profile.js`'s `reconstructKeyPairFromRaw`), so the scalar can't be read off it directly for an already-unlocked session. The setup flow instead re-derives it via `exportRawIdentity(profileId, passphrase)` (the same re-decrypt-under-passphrase pattern `btn-link-device`/device-linking already uses) → `importPrivateKeyRaw(..., extractable=true)` → `exportPrivateKeyScalar`. This works identically for a portable account or a plain permanent profile — passphrase re-entry is a deliberate confirmation step for this high-impact action, not a workaround for a gap.
  - **QR code deferred**: only the text-export path (`encodeShareAsText`/`decodeShareFromText`, `spirit-share:<x>.<threshold>.<total>.<y-base64url>`) is implemented. No QR-rendering library exists anywhere in this codebase yet (checked `client/js/vendor/` and `client/index.html`), and vendoring one is a separate decision this section deliberately does not make. The UI shows the text string with an explicit "QR not yet supported" hint.
  - **"Contact must eventually reconnect" limitation**: a selected contact who is offline at setup time and never reconnects while queued never receives their share (the outbox has no expiry/retry beyond "next time this contact's identity is verified"). Accepted for a first cut — re-running setup re-splits and re-queues anyway, and there is no persistent broadcast channel (zero-database invariant) to do better without a bigger design change.
  - **Re-split staleness limitation**: `trustedShares.js` stores at most one share per `ownerFingerprint`, OVERWRITTEN on every new announce — a fresh `splitSecret` call produces an incompatible `(threshold, totalShares)` polynomial set, so mixing shares from two different splits does not reconstruct anything. Exec review iter1 additionally caught and fixed a related race: the immediate-send path (contact is the live peer at setup time) now also dequeues any stale outbox entry for that same contact, so a contact can never end up holding an older queued share that overwrites the fresh one they were just sent live.
- [x] **Exec review**: iter1 — [reviews/social-recovery-S2-iter1.md](../reviews/social-recovery-S2-iter1.md). One must-fix (stale-queue/immediate-send race, fixed: added `dequeueRecoveryShareForContact`) and one nice-to-have (`x` not bounded by `totalShares` on receive, fixed) found and resolved in iteration 1; one nice-to-have (received `y` length not checked against 32 bytes) explicitly deferred to S3 as low-priority/cosmetic. Converged after iter1 — no second review round needed, fixes were small and directly addressed the reviewer's exact quoted code.

## Секція S3: Відновлення (recovery flow)

Дві симетричні половини, обидві потрібні для реально робочого флоу "з нуля до відновленого акаунта":

1. **Бік довіреної особи (trustee)**: у людини, що зберігає чужу частку в `trustedShares`, досі НЕМАЄ способу її переглянути/дістати назад, коли власник акаунта просить допомогти з відновленням (S2 лише зберігав отримане, без UI перегляду). Потрібен невеликий список "Частки відновлення, які я зберігаю" (fingerprint власника + кнопка "Показати як текст", повторно використовує `encodeShareAsText` з S2).
2. **Бік власника акаунта, що відновлюється**: новий UI-флоу на екрані «Акаунт» (поруч із наявними мнемонікою/keyfile/portable-login) — "Відновити через довірених контактів": вставка `threshold` текстових рядків (`spirit-share:...` формат з S2, отриманих поза застосунком від довірених осіб), валідація узгодженості (`threshold`/`totalShares` збігаються між усіма вставленими частками — інакше вони з різних split-циклів і не комбінуються), `combineShares` → `importPrivateKeyFromScalar` → той самий шлях адопції ідентичності, що вже існує для мнемоніки/keyfile (`adoptScalarIdentity(scalarBytes, localPassphrase)` з Фази 3, `client/js/profile.js`) — повторне використання наявного коду, не новий шлях.

QR-сканер свідомо НЕ входить у цю секцію (той самий дефіцит vendored-бібліотеки, що й QR-рендеринг у S2) — лише текстовий ввід/вивід, симетрично до S2's текстового експорту.

## Верифікація

Кожна секція — окремий spec-first/test-first цикл. S1 (чисте крипто-ядро) — безпечно почати без додаткових питань, той самий паттерн, що P2a/PN1. S2/S3 (реальний розподіл часток на чужі пристрої, реальний UI відновлення) торкаються моделі довіри глибше — перед стартом S2 варто ще раз коротко звірити з користувачем деталі зберігання (`trustedShares` окремим сховищем) і UX вибору N/k, якщо design-питання виникнуть під час імплементації.
