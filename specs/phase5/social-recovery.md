# Спека: Соціальне відновлення акаунта через Shamir secret sharing (Секція P4)

Розширення бектрогу `specs/phase5/security-hardening.md`'s Секції P4 ("Соціальне відновлення портативного акаунта"). Рішення узгоджені з користувачем 2026-07-18 через AskUserQuestion.

## Контекст: що саме втрачається без цієї функції

Дослідження перед дизайном (2026-07-18) показало: у поточній Argon2id-схемі (Фаза 3, `specs/phase3/deterministic-accounts.md`, `client/js/deterministicIdentity.js`) втрата портативного пароля НЕ знищує зашифрований блоб — вона знищує єдиний спосіб детерміновано відтворити 32-байтний скаляр identity private key на пристрої без локальної копії профілю. Той самий 32-байтний скаляр — це вже той секрет, що кодується в мнемоніці/keyfile (`docs/accounts.md`). Отже: втрата і мнемоніки/keyfile, і локального пристрою з профілем в IndexedDB = незворотна втрата ідентичності, незалежно від того, портативний акаунт чи ні.

## Рішення (узгоджено з користувачем)

1. **Що розщеплюємо**: сам 32-байтний скаляр identity private key (`client/js/deterministicIdentity.js`'s вихід перед `importPrivateKeyFromScalar`) — той самий секрет, що вже кодується мнемонікою/keyfile. Відновлює доступ незалежно від того, чи акаунт портативний, чи звичайний перманентний профіль без збереженого backup.
2. **Як частки (shares) потрапляють до довірених контактів**: ОБИДВА канали підтримуються паралельно (той самий згенерований набір часток, два способи його роздати):
   - **Автоматичний P2P-канал**: під час налаштування відновлення користувач обирає N вже верифікованих контактів зі свого списку; кожен обраний контакт отримує свою частку зашифрованою через наявний E2EE-канал (той самий патерн, що й `device-list-announce`/`push-subscription-announce`) і зберігає її у власній IndexedDB, доки власник акаунта (чи сам довірений контакт) не видалить.
   - **Ручний QR-код/текстовий експорт**: користувач також може згенерувати ту саму частку як QR-код чи текстовий рядок і передати довіреній особі будь-яким іншим каналом (голосом, особисто, іншим месенджером) — жодна частка тоді не проходить через P2P-канал Spirit і не зберігається автоматично на нічиєму пристрої.

## Нова властивість моделі довіри (явно визнана, не прихована)

Досі жоден чужий пристрій ніколи не зберігав фрагмент чужого секрету — контакти зберігали лише публічну інформацію (fingerprint, device list, proof set, push-підписку). Ця секція вперше вводить: **фрагмент чужого приватного ключа, що зберігається на пристрої довіреного контакту**. Це свідомий, пороговий (k-з-n) компроміс: одна окрема частка нижче порогу k криптографічно марна (інформаційно-теоретична гарантія Shamir secret sharing — не потребує додаткового шифрування частки самої по собі, хоча канал доставки і сховище на боці контакту все одно варто захищати як звичайні секрети профілю). Компрометація одного (< k) довіреного контакту НЕ розкриває identity-ключ власника.

## Секція S1: Shamir secret sharing — чисте крипто-ядро (GF(256), без інтеграції)

Той самий підхід, що вже виправдав себе для `ratchet.js` (P2a) і `webPushCrypto.js` (PN1): спочатку самодостатній, повністю протестований крипто-модуль без залежності від решти застосунку, окремий exec review, і лише ПОТІМ (наступні секції) — розподіл часток і UI відновлення.

Без bundler-інваріант: реалізація на чистому Web Crypto API де можливо (`crypto.getRandomValues` для випадкових коефіцієнтів полінома), і власна арифметика GF(2^8) (той самий скінченний field, що й AES S-box/Reed-Solomon QR-кодів) — не потребує зовнішньої бібліотеки, бо GF(256) арифметика тривіальна (додавання = XOR, множення через log/antilog-таблиці).

- [ ] **Tests**: `client/tests/shamir.test.js` (новий файл) — `splitSecret(secretBytes, {threshold, shares})` повертає `shares` часток, кожна відновлює точно ті самі байти при комбінуванні БУДЬ-ЯКИХ `threshold` з них (не обов'язково перших); менше ніж `threshold` часток НЕ відновлює правильний секрет (і не кидає підказку про правильність — детектована лише зовнішньою перевіркою, напр. чи відновлений скаляр валідний P-256 приватний ключ); `combineShares(shares)` детермінований і не залежить від порядку часток; коректна робота для реалістичного розміру секрету (32 байти, розмір identity-скаляра); межові випадки (`threshold > shares` — помилка; `threshold < 2` — помилка, немає сенсу розщеплювати на "поріг 1"; порожній/некоректний вхід).

## Верифікація

Кожна секція — окремий spec-first/test-first цикл. S1 (чисте крипто-ядро) — безпечно почати без додаткових питань, той самий паттерн, що P2a/PN1. S2/S3 (реальний розподіл часток на чужі пристрої, реальний UI відновлення) торкаються моделі довіри глибше — перед стартом S2 варто ще раз коротко звірити з користувачем деталі зберігання (`trustedShares` окремим сховищем) і UX вибору N/k, якщо design-питання виникнуть під час імплементації.
