# Спека: Read-only адмін-панель параметрів вузла (`/#/server`)

Показ операційних параметрів сигнального вузла (rate limits, CORS, session TTL тощо) на екрані «Сервер», захищений окремим admin-паролем. **Лише перегляд** — редагування залишається виключно через FTP/`config.php` (рішення користувача, спрощений обсяг).

Джерело істини: `docs/signaling-protocol.md`, `server/config.php`.

## Технічні рішення (узгоджено з користувачем)

- **Право доступу = право редагувати `config.php`**: окремий admin-пароль, який задає той, хто має FTP-доступ (той самий актор, просто без прямого перевикористання FTP-секрету — змішувати рівні довіри хостингу й веб-застосунку не можна).
- **Read-only**: жодного `set_config`, жодного запису — усуває весь клас ризиків (code injection, config corruption, потреба валідації на запис).
- **Токен, не сесія**: `admin_login` перевіряє пароль (`password_verify` проти bcrypt-хешу в конфізі), повертає HMAC-підписаний токен з TTL 15 хв (окремий секрет підпису `ADMIN_TOKEN_SECRET`, не сам хеш пароля) — узгоджено зі stateless-дизайном вузла (жодних PHP-сесій).
- **Білий список полів для показу**: rate limits, session TTL, max sessions, allowed origins, global access (bool), fetch_proof (bool) + його ліміти. **Виключено**: шляхи файлів (`DB_FILE`/`RATE_LIMIT_FILE`/`LOCK_FILE` — розкриття структури файлової системи), `WHITE_LIST` (може містити чужі публічні ключі), сам хеш пароля й секрет підпису.
- **Фіча вимкнена за замовчуванням**: якщо `ADMIN_PASSWORD_HASH` відсутній/порожній у `config.php` — обидві дії повертають 403 "адмін-доступ вимкнено на цьому вузлі" (той самий патерн, що й `ENABLE_PROOF_PROXY`).
- **PHP-секції без локальних тестів** (established waiver для `server/`, PHP/Composer недоступні локально) — верифікація живим деплоєм + curl на kolomedi/kibr.
- **Токен лише в пам'яті клієнта** (не localStorage) — повторний логін при новій сесії/оновленні сторінки.

## Секція S1: Серверні дії `admin_login` / `admin_get_config`

- [x] **Tests**: — (PHP waiver, верифікація живим curl на kolomedi ТА kibr, ідентичні результати обох): правильний пароль → 200 + `{token, expires_at}`; неправильний → 401; `admin_get_config` з дійсним токеном → 200 + config-поля БЕЗ жодного забороненого ключа (`DB_FILE`/`RATE_LIMIT_FILE`/`LOCK_FILE`/`WHITE_LIST`/хеш/секрет); без токена → 401; з підробленим токеном → 401.

## Верифікація

S1 — жива перевірка на реальних вузлах (curl, як усі попередні PHP-секції). S2 — test-first (jsdom), потім жива перевірка в preview + деплой на обидва хости.
