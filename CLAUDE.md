# Spirit — інструкції для Claude (будь-який акаунт)

Цей файл читається автоматично Claude Code при роботі в цій директорії, незалежно від того, з якого акаунта підключився волонтер. Він існує саме для того, щоб дисципліна розробки не залежала від особистих налаштувань конкретної людини.

## Модель співпраці ("Claude Country")

Spirit розробляється кількома незалежними волонтерськими Claude-акаунтами, що працюють над однією кодовою базою на GitHub у межах власних кредитів — без спільної розмови чи прямої координації між собою. Єдиний канал комунікації між агентами — **сам репозиторій**: git-історія, `specs/`, `specs/reviews/`, `docs/roadmap.md`.

**Якщо ти читаєш це, підключившись до задачі вперше (або перед стартом будь-якої нової секції)**:
0. **Синхронізуйся з GitHub ПЕРШ НІЖ щось починати**: `git fetch origin` + `git log origin/main --oneline -10` (чи `git pull`, якщо локальна гілка чиста) — перевір, чи інший волонтерський Claude-акаунт вже не допрацював цю чи суміжну секцію, поки тебе не було. Працювати від застарілого знімку репо — гарантований конфлікт з паралельним волонтером. (`main` — єдина робоча гілка проєкту з 2026-07-18; `master` синхронізовано в `main` і надалі не використовується окремо.)
1. `git log --oneline -30` і `docs/roadmap.md` — що вже зроблено, що в бектрозі.
2. `specs/` — знайди спеки з незавершеними чекбоксами (`grep -rn "\- \[ \]" specs/`). Кожна незавершена секція — незалежна одиниця роботи.
3. `specs/reviews/` — історія exec-review, щоб зрозуміти вже прийняті рішення й дисміснуті знахідки (не піднімай їх знову без нових доказів).
4. Візьми ОДНУ секцію (не всю спеку одразу, якщо вона велика) — заверши її повним циклом (тести → імплементація → review → тік чекбоксів → коміт → пуш → деплой+перевірка на обох тестових хостах) перш ніж переходити далі. Якщо не встигаєш завершити секцію в межах сесії — залиш git-стан чистим (закомічена робота лише для повністю завершених і review-пройдених секцій), а спеку — з чітким записом, що зроблено, а що ще ні.

## Інженерна дисципліна (обов'язкова, без винятків)

1. **Spec-first** — для будь-якої зміни, що торкається 3+ файлів: спека в `specs/<область>/<назва>.md` з трійками Tests/Impl/Exec-review на секцію. Дрібні зміни (<3 файлів, one-line фікс) — без повної церемонії, одна фраза підтвердження в чаті достатньо.
2. **Test-first (жорсткий гейт)** — падаючий тест ДО будь-якої зміни імплементації. Підтвердь RED, потім імплементуй. Тест і імплементація — в одному коміті.
3. **Exec-review** — після зеленого коду секції (чи логічної групи секцій) заплануй незалежний рев'ю (інший Claude-агент/акаунт чи Task-subagent з моделлю Opus, якщо доступна) диффу. Реальні знахідки — виправляй або відхиляй з конкретним доказом (не "здається, ок"). До 3 ітерацій на секцію.
4. **Commit gate** — коміт лише після зеленого тесту + review, що зійшовся. Один коміт = одна повністю опрацьована секція спеки (чи невелика логічно пов'язана група секцій, якщо вони прогнозовано ревьюяться разом).
5. **Push gate**: пуш на GitHub — за умовчанням АВТОМАТИЧНО одразу після завершення модуля/секції чи іншої логічної відсічки (коміт із зеленим тестом + зійшовшимся review), без додаткового підтвердження в чаті — це стандартна практика проєкту в моделі Claude Country (інші волонтерські акаунти читають прогрес лише з GitHub, тому недопушений коміт для них не існує). Виняток: якщо конкретна людина в поточній сесії явно попросила НЕ пушити (наприклад, працює над спірним/незавершеним шматком) — тоді дотримуйся звичайного гейту й чекай на пряме "push"/"go".
6. **Деплой і жива перевірка**: клієнтський код (`client/`) розгортається FTP-скриптом (тимчасовий Node-скрипт з `basic-ftp`, credentials з `deploy/*.local.netrc` — НІКОЛИ не в git) на обидва хости — `spirit.kolo.media` (kolomedi) і `spirit.kibr.com.ua` (kibr) — одночасно, після кожного коміту/пушу, за умовчанням без додаткового підтвердження (стандартна практика проєкту). Після завершення модуля чи логічної відсічки — не лише задеплоїти, а й **перевірити функціонал живо на обох хостах** (браузер-агент чи ручна перевірка ключового флоу), а не лише покластися на unit-тести. PHP-код (`server/`) розгортається в піддиректорію `spirit/` на кожному хості (НЕ в корінь — `server/public/index.php` шукає `../config.php`, тобто `spirit/config.php`, не `config.php` в корені сайту).
7. **Секрети**: `deploy/*.local.*` (FTP-креденшли) і `server/config.secrets.php` (admin-пароль, `ENABLE_PROOF_PROXY`) — обидва в `.gitignore`, ніколи не комітяться. Якщо секрет потрібен для роботи, а локальної копії немає — попроси людину надати її поза git.

## Архітектурні інваріанти (не порушувати без явного обговорення з людиною)

- **Zero-database** (D1): сигнальні вузли не зберігають нічого постійного понад ефемерні сесії/кімнати з TTL. Будь-яка ідея "зберігати повідомлення на сервері, поки одержувач офлайн" — архітектурна зміна, що потребує явного погодження, не мовчазної імплементації.
- **E2EE за замовчуванням** (докладно — `docs/e2ee.md`).
- **Без bundler**: клієнт — чисті ES-модулі, деплояться напряму по FTP. Нові npm-залежності для браузерного коду мають бути vendored як self-contained ESM-файл у `client/js/vendor/` (див. `client/js/vendor/hash-wasm.esm.js` як приклад) — не `import` з `node_modules` напряму.
- **Кросплатформність**: увесь код (крипто включно) має працювати в браузері й на мобільному однаково — жодних Node-only чи native-залежностей у `client/`.

## Поточний стан і пріоритети

Дивись `docs/roadmap.md` для повної картини фаз. Станом на 2026-07-13: Фази 0-4 завершені (MVP, профілі/мультипристрій/backup, verification/OAuth, портативні акаунти, ефемерний режим). **Фаза 5 (`specs/phase5/security-hardening.md`) — головний бектрог, не почато.** Три пріоритети (записано за прямим запитом користувача, впорядковано за впливом на реальну модель загроз):

1. Метадані-стійкий транспорт (форсований TURN, власний STUN, опційний Tor).
2. Ratchet поверх наявного ECDH-хендшейку (post-compromise security).
3. Encrypted push notifications (повідомлення, коли вкладка закрита, без порушення zero-database).

Секції P1(в) і P3 у цій спеці явно вимагають обговорення з користувачем (через питання в чаті) ПЕРЕД кодом — вони змінюють модель довіри проєкту, не лише додають функціонал.
